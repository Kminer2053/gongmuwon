# 릴리스 수용 테스트 계획 (Release Acceptance Plan) v1

목적: 설치패키지 최초 설치부터 실사용까지의 완성도를 **사전 수립된 정량 기준**으로 측정한다.
본 문서는 1회용이 아니라 **수정 때마다 재실행하는 평가지표(스코어카드)** 다. 결과는
`docs/qa/scorecards/YYYY-MM-DD.md` 로 누적해 완성도 추세를 추적한다.

## 0. 반복 규칙 (전 시나리오 공통)
- 기본 3회 실행. 3/3 통과 → PASS(안정). 1회라도 실패 → 5회로 확장, 성공률 기록(4/5 이상=CONDITIONAL, 미만=FAIL).
- 직전 스코어카드에서 2회 연속 PASS(안정)한 TC는 이번 회차 1회 실행으로 축소 가능(회귀 감시).
- 실행 도구: 설치 단계=수동+로그/증거 수집(RUN_FULL_VALIDATION.bat), 앱 기능=Playwright 헤드리스(스크립트 재사용) + 사이드카 API 검증 병행.
- LLM 응답 판정: 결정적 기준(키워드 존재·출처 수·형식 마커) 우선. 주관 판정 필요 시 3회 중 2회 합의.

## 1. 사전 검증기준 (품질 게이트)
| 게이트 | 기준 | 측정 |
|---|---|---|
| G1 설치 | 클린 환경에서 AI팩 설치 완주, evidence `ready:true`, 전 체크 PASS | evidence.json |
| G2 첫실행 | 앱 기동 ≤10s, LLM 연결 표시, 온보딩 정상 | Playwright |
| G3 색인 | 실폴더(2026_AI혁신처) 533→색인 문서 ≥410, 실패 0, 추출 품질 평균 ≥0.75 | DB 집계 |
| G4 검색신뢰 | 자연어 8문 벤치 ask ≥7/8, 키워드 8/8 (자연어 질의셋은 §3-D 표로 고정) | askbench 스크립트 |
| G5 대화연계 | 연계 TC(§3-B) 전항 PASS, 스킬 턴 ≤1s·LLM 턴 ≤60s | Playwright+API |
| G6 위키품질 | 다문서 주제 종합 성공률 ≥90%, 종합 페이지 골격(개요·핵심내용≥2·근거각주) 충족 100% | 전수 스크립트 |
| G7 문서품질 | 4양식 구조 생성: 위계기호 정확 100%, 지시 수치 보존 ≥90%, placeholder 0 | 반복 생성 |
| G8 회귀 | 사이드카 pytest·데스크톱 vitest 전체 그린 | CI 스위트 |
종합 점수 = 게이트별 (PASS=1, CONDITIONAL=0.5, FAIL=0) 가중합 ×100 / 8. **출시 기준 ≥ 90.**

## 4. 산출물·재사용
- 실행 스크립트 보관: `docs/qa/scripts/`(askbench.mjs, feature-e2e.mjs, chat-link.mjs, wiki-usage.mjs — 기존 E2E 스크립트 정리 이관)
- 스코어카드 양식: 게이트표 + TC별 (회차/통과/실패로그 링크) + 종합점수 + 직전 대비 증감.
- 수정 릴리스마다: 변경 영향 게이트는 전량, 나머지는 축소 규칙(§0) 적용.

3) 게이트 미달 항목 → 백로그 등록 → 수정 → 해당 게이트 재실행
