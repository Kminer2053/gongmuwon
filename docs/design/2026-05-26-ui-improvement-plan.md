## 1. 프로젝트 목적 / 목표 사용자경험 (UX 진단의 전제)

- **사용자**: 공공기관 사무업무자. 공문/보고서/일정/지식자료가 폐쇄망 PC 안에서 흩어져 있고, 외부 SaaS 사용에 제약이 있다.
- **핵심 가치**: 업무대화 세션을 허브로 일정·파일·지식폴더·문서작성·실행기록을 묶는 "개인 업무 기억 시스템".
- **성공 신호**:
  - 앱을 켜고 3초 안에 "지금 무슨 일을 할 수 있는지" 보인다.
  - 모델이 어떤 근거(지식/연결파일/이미지/도구)를 써서 답했는지 답변과 같이 보인다.
  - HWPX 산출까지 가는 흐름이 "문서작성 한 화면 안에서" 끊기지 않는다.
- **현재 위치**: MVP 기능은 모두 들어가 있고 2차 피드백까지 반영되어 있다. 남은 작업은 **화면 밀도 정리, 정보 위계, 상태 가독성**이다.

## 2. 디자인 진단 요약 (design-critique 관점)

### 2.1 글로벌 셸 (Global Shell)

| 관찰 | 진단 |
| --- | --- |
| 상단 브랜드 카드가 모든 화면에서 ~110px 차지 (eyebrow + h1 + tagline) | 같은 영역에서 페이지 칩, 줌 컨트롤, 우측 패널 토글, 새로고침, 신호등이 한 줄 더 있어 **상단 정보 위계가 6단계**로 늘어남 → 본문 영역이 좁아짐 |
| 좌측 패널이 3x3 아이콘 그리드(7개)+ 대화세션 리스트 | 대화세션 리스트는 **업무대화 화면에서만 의미**가 있는데 모든 화면에서 좌측 절반을 차지함 |
| 우측 컨텍스트 패널이 "선택 상태 / 선택 일정 / 선택 세션 / 선택 ReferenceSet" 라벨-값 카드 6개로 고정 | 화면이 바뀌어도 같은 내용 → **컨텍스트 패널이 컨텍스트를 안 따라감** (정의에 반함) |
| 우측 패널 상단 6개 탭 아이콘이 24px·라벨 없음 | 호버 없이는 무엇인지 알 수 없음. WCAG 2.1 명명된 컨트롤 기준 미달 |

### 2.2 업무대화

| 관찰 | 진단 |
| --- | --- |
| 헤더 아래 "연결 일정 / 선택 일정과 연결 / 파일 연결 / 문서작성으로 이어가기 / 연결 파일 0개 / 이 세션 지식 반영" 6 컨트롤이 가로 한 줄 | 메시지 스크롤과 입력 사이에 끼어 **읽기-쓰기 흐름을 끊음**. 비슷한 무게의 버튼/칩이 섞여 어떤 게 토글이고 어떤 게 액션인지 구분 어려움 |
| 사용자 메시지 = 검정 단색 라운드 풀필 (max-width ~70%) | 라이트 톤 UI에서 너무 무거움. 콘트라스트 대비는 14:1로 좋지만 시각적 무게 편중 |
| 어시스턴트 메시지에 `gongmu-skill / intent.plan, schedule.create, knowledge.search` 한 줄 + 우측 끝에 `응답 11.6초` 알약 | 두 메타 정보가 한 묶음으로 안 묶여서 시선이 좌→우로 점프. 첨부/검색/도구 사용 여부도 같이 보였으면 좋겠음 (P1 audit 항목과 일치) |
| 입력창이 1-line처럼 보이는 텍스트영역, 보내기 버튼이 회색 | 라이트 톤에서 메인 액션 버튼이 secondary처럼 보임. 글로벌 ".button"이 검정 단색인데 보내기 클릭 직전 disabled-look이 자주 발생 |

### 2.3 일정

| 관찰 | 진단 |
| --- | --- |
| 주간 보기가 **시간축이 없는** "요일별 카드" 리스트 | 캘린더 메타포 깨짐. 같은 시간대 이벤트 겹침을 표현 못함. 이벤트 위치=시간이라는 시각적 단서 부재 |
| "월/주/일" 토글 (상단) + "보기" 셀렉트 (하단 폼) 중복 | 같은 상태를 바꾸는 컨트롤이 두 군데 |
| "선택 칸 일정 입력 / 아직 선택 없음" 우측 텍스트 | 폼이 비활성인지 활성인지 색·상태 변화로 표현 안 됨 |

### 2.4 내 지식폴더

| 관찰 | 진단 |
| --- | --- |
| 상단에 큰 지식그래프 카드 + 그래프 안 "전체/폴더/문서/키워드" 필터 + 그래프 밖 좌측 카드에 통계 칩(폴더 4 / 문서 23 / 본문 21 / 대화기록 2 / 메타데이터만 2) | 같은 정보 도메인(통계)이 두 카드에 나뉘어 있음 |

## 4. 디자인 시스템 진단 (design-system)

### 4.1 토큰 불일치

| 토큰 | 현재 값 | 사용처 | 진단 |
| --- | --- | --- | --- |
| `border-radius` | 24px / 22px / 18px / 16px / 14px / 999px (6단계) | 카드 24 / feature-rail 22 / topbar-current 18 / 일부 16 / 버튼 14 / 칩 999 | **6 스케일 → 3 스케일**(12/20/full)로 줄이기 권장 |
| `--bg-panel` | `rgba(255,255,255,0.94)` + blur(10px) | 거의 모든 카드 | 카드 밀도가 높은 상황에서 blur는 GPU 부담 + 시각적 노이즈. 핵심 패널만 유지 |
| 노란 info 색 | 5개 화면에 같은 노란 톤이 다른 의미(연결 안내 / 카운트 칩 / 정책 안내 / 응답시간 / 가까운 일정)로 쓰임 | "노란색은 X다"라는 일관 의미가 없음 |
| 강조 액션 | 검정(`#111214`) 솔리드 버튼 단일 | 대비는 좋지만 brand 색(`rgba(31,107,85)` = 진한 그린)이 좌측 메뉴 활성 상태에만 사용. **brand 비대칭** |

### 4.2 컴포넌트 중복

`apps/desktop/src/styles.css`에서 grep:

| 패턴 | 발견 횟수 | 비고 |
| --- | --- | --- |
| `border-radius: 24px` | 21회 | 카드 패턴 |
| `box-shadow: var(--shadow)` 사용 카드 | 18회 | 거의 모든 카드가 같은 shadow → 깊이 단계 없음 |
| `padding: 0.9rem 1.05rem` 같은 거의 동일 값 | 12회 | 토큰화되지 않은 micro-padding |

### 4.3 결론

디자인 시스템이 **존재하지만 토큰화되어 있지 않음**. CSS 변수 6개(`--bg-panel`, `--line-soft`, `--accent` 등)는 잘 잡혀 있는데, 그 변수를 거치지 않고 raw 값(24px, rgba(31,107,85,0.12) 등)이 곳곳에 반복되어 있다. 시안은 토큰 일치 후 어떻게 보일지를 함께 제안한다.
