# 위키 백과사전화 재설계 (Wave D) — 지식그래프 메모리 방법론 이식

참고: 사용자 제공 스레드(@ur.future.ai)가 소개한 "AI가 스스로 작성·관리하는 지식 시스템"
오픈소스 계열(Graphiti·Zep, Mem0, Letta, Cognee, GraphRAG류)의 공통 방법론을
로컬-우선·경량모델 제품에 맞게 이식한다. 목표: **주제 위키를 링크 스텁에서
나무위키식 백과사전 페이지로** (2026-07-11 백로그 F-08/F-09/F-10/F-02/F-11).

## 이식하는 5가지 방법론 → 설계 매핑

| 원천 방법론 | 우리 설계 |
|---|---|
| ① Entity Resolution (Graphiti/Zep — 중복 엔티티 해소) | **주제 정규화**: 보강 시 LLM에 '기존 주제 사전'을 제공해 재사용 유도 + 정규화 키(공백·조사 제거) 병합 + 슬러그 버그 수정 |
| ② Community Summaries (GraphRAG — 클러스터 요약) | **백과사전 종합**: 문서≥2 주제는 추출본 발췌를 모아 LLM이 종합 서술 생성 |
| ③ Temporal Graph (Graphiti — 사실의 시간 추적) | **경과·연표 섹션**: 문서 날짜(파일명 날짜>mtime)로 타임라인, 버전 가족은 최신본 기준 서술+구판은 연표로 |
| ④ Episodic→Semantic 증류 (Mem0/Letta — 증분 요약) | **증분 재종합**: 신규/변경 문서 색인 시 해당 주제만 dirty 마킹 → 다음 보강에서 그 주제만 재종합(전량 재생성 금지) |
| ⑤ Self-building Pipeline (Cognee cognify) + Provenance | **자동 연속 보강** + 모든 종합 서술에 **각주 출처**(근거 문서 링크) |

## 백과사전 주제 페이지 골격 (나무위키식)

```markdown
# {주제}
> 한 줄 정의 (개요 첫 문장)

## 개요
{2~4문장 정의·맥락 — LLM 종합}

## 핵심 내용
### {소주제1}
{문서 내용 종합 서술} [1][2]
### {소주제2} ...

## 경과
- 2026-01-13 처장회의 발표 [3]
- 2026-05-22 정기회의(6차) 결과 [4]

## 문서별 요점
- **[문서제목1]** — 핵심 1줄 [1]
- ...

## 연관 주제
- [[주제A]] · [[주제B]]

## 근거 문서
[1] 문서카드 링크 · 원본 파일명
```

## 구현 사양

1. **주제 정규화 (F-08)** — `knowledge_wiki.py` 보강 경로:
   - LLM 주제 부여 프롬프트에 기존 주제 상위 N(빈도순 40개) 제공: "가능하면 아래 기존 주제를 재사용".
   - 저장 전 정규화: NFC·공백정리·`_strip_josa_suffix`·소문자(ASCII), 정규화 키가 같으면 기존 표기로 통일.
   - 슬러그: `wiki_slugify(topic)` 한 번만(이름 중복 결합 버그 수정), 후행 공백 제거.
2. **백과사전 종합 (F-09)** — 신규 모듈 `topic_synthesis.py`:
   - 대상: 문서 ≥2 주제(싱글턴은 종합 생략 — 문서 카드 링크만).
   - 입력: 주제 소속 문서들의 (제목·날짜·요약 or 추출본 앞 500자) — 최대 8건, 컨텍스트 예산 내 절단.
   - 출력: 위 골격 JSON(개요/핵심내용[소주제·서술·근거번호]/경과/문서별요점/연관주제) → pydantic 검증·1회 repair(문서작성 파이프라인과 동일 패턴).
   - 렌더: `_write_topic_pages`가 종합본 있으면 백과사전 골격으로, 없으면 기존 링크 목록. front matter에 `synthesized: true`, `synthesized_at`.
   - 트리거: enrich 잡 말미에 dirty 주제 종합(주제당 LLM 1회). 문서 증분 색인 시 해당 주제 dirty 마킹(DB `topic_dirty` 테이블 또는 topics_json 변경 감지).
3. **자동 연속 보강 (F-11)**: enrich 잡 완료 시 remaining>0이면 다음 배치 잡을 자동 체이닝(동일 resource_key 직렬). 취소 버튼 유지, 중단 복구는 "재개"로 표기. 대시보드 'LLM 요약 보강' 카드에 커버리지(요약 보유 n/전체) 표시.
4. **허브 개선 (F-10)**: 개요에 LLM 3~4줄 업무 요약(보강 시 생성·캐시), '핵심 문서' 선정에서 doc_role=양식/서식 제외+보고서·계획 우선, 문서 행 표시는 제목만(원본 절대경로는 링크 title 속성으로).
5. **제목 규칙 (F-02)**: 본문유래 제목이 저품질(≤3자·날짜만·`Part N` 조각·비한글 잡음)이면 파일명(stem) 사용. 주제 슬러그와 함께 단위 테스트.
6. **프론트**: `TopicsWikiTemplate`가 종합 섹션(개요 하이라이트 + 핵심내용/경과/문서별 요점 마크다운)을 관련 문서 카드 위에 렌더. 근거 각주는 문서 카드로 링크.

## 경량모델(gemma-4-E2B) 대응 원칙
- 지능은 빌드타임에: 종합은 백그라운드 보강에서 1주제 1호출, 입력은 요약·발췌로 압축(≤2.5k자).
- 실패 허용: 종합 실패 주제는 링크 목록 유지 + 다음 보강에서 재시도(잡 실패로 승격 금지).
- 결정적 부분(정규화·연표·각주 번호)은 코드로, 서술만 LLM.

## 수용 기준 (실폴더 재검)
- 주제 수 감소(정규화 후 신규 보강분 기준 싱글턴 비율 유의미 하락 — 재보강 표본에서 확인).
- 문서≥2 주제 표본 5개: 백과사전 골격(개요+핵심내용+근거 각주) 렌더 확인.
- 허브 표본: 개요 요약 존재·양식 파일이 핵심 문서에서 제외.
- 기존 pytest/vitest 전체 그린.
