# 4호 라운드 UX 개선 계획 — 2026-07-15

사용자 개선요구 5건에 대한 조사·설계 확정본. 트랙별 상세(현행 구조·설계·파일별 변경·검증기준·시나리오·리스크)는 `2026-07-15-fix-tracks/*.json` — **구현자는 반드시 해당 JSON을 읽고 그대로 실행**한다. 조사는 5개 병렬 에이전트가 코드 실측(파일:라인 인용)과 실DB(읽기 전용) 집계로 수행했다.

## 0. 실행 원칙 (2·3호와 동일, 상시 적용)

1. **검증기준 사전 확정**: 각 트랙 JSON의 `verification_criteria`가 구현 전 확정된 PASS 기준이다. 구현 중 임의 완화 금지 — 완화가 필요하면 근거를 계획서에 비고로 남긴다.
2. **수정 전 FAIL 재현**: 동작 변경 트랙(T2·T3·T4·T5)은 신규 테스트를 먼저 작성해 현행 코드에서 FAIL함을 확인한 뒤 수정한다(T1은 신규 기능 성격이라 신규 테스트 선작성만).
3. **시나리오 커버리지**: 트랙당 8~9개 시나리오(네거티브 컨트롤 각 2개 포함)를 전부 결정적 테스트(vitest/pytest/cargo test)로 구현. LLM 변주가 없는 순수 UI/렌더 변경이므로 3×반복은 불요 — 대신 결정적 판정을 전 시나리오에 강제.
4. **정직 보고**: 실패·보류·기준 미달은 그대로 보고.
5. **의도된 계약 변경의 명시적 처리**: 이번 라운드는 기존 테스트 어서션을 의도적으로 바꾸는 지점이 많다(각 트랙 JSON `risks`에 전수 명시). 계약 갱신 커밋과 기능 커밋을 같은 커밋에 묶어 G8이 중간에 레드가 되지 않게 한다.

## 2. 구현 순서와 충돌 관리

T3·T4·T5가 모두 `KnowledgeScreen.tsx`(5,260줄)를 수정하므로 **순차 진행** (병렬 금지):

1. **T2** (사이드카 hwpx_writer 중심 — 프런트는 DocumentsScreen 미러 1곳, 독립)
2. **T1** (store.tsx + main.rs — 독립)
3. **T5** 설정 탭 재구성 → **T3** 위키 검색 분리 → **T4** 트리 그룹핑 (동일 파일 순차, 각 트랙 완료 시점마다 vitest 전체 그린 확인 후 다음 착수)

트랙 JSON의 라인 번호는 조사 시점 기준 — 선행 트랙 반영 후 어긋나므로 앵커 문자열(testid·함수명)로 재탐색할 것.

## 3. 게이트 계약 변경 (의도됨 — 다음 수용 라운드 판정 기준)

- **G7(문서품질)**: 1p 요약은 글상자(표) 내 ◦ 항목으로 렌더되며 `□` 부모 없이 허용. 평문 미리보기의 `─` 룰 라인은 위계기호 판정 제외. → `docs/qa/release-acceptance-plan.md` G7 비고에 명문화(T2 커밋에 포함). fullReport는 현행 '□ 요약' 유지(네거티브 컨트롤).
- **G8(회귀)**: 의도적 어서션 갱신 지점 — menu-zoom 2건(100%→90%), documents-authoring 4곳+test_document_authoring 1곳(요약 표기), knowledge-sources 3구획(토글→분리 UI), openSettingsTab 헬퍼 3파일(서브내비 클릭 추가). 이외 전부 무수정 그린이 요구사항.

## 4. 검증 계획 (사전 확정)

- 트랙별 `verification_criteria` 전 항목 PASS (합계 38개 기준, 41개 시나리오).
- 결정적 우선: HWPX는 zip 해부(`hp:tbl` 내 문장 존재 판정), UI는 vitest DOM/testid, Rust는 `menu_zoom_target` 단위 테스트, 실DB 스모크는 읽기 전용 GET.
- LLM 배터리 재실행 불요 판단: 검색 API(`search()`)·ask 경로·프롬프트 무변경(T3는 프런트 재배치, T4는 wiki_tree per-item 필드 추가만). 단, T2가 콘텐츠베이스 .md에 룰 라인을 추가하므로 **G7 문서생성 5종 스모크**(1p 3건 + fullReport/officialMemo 각 1건)는 설치본에서 실측.
- 설치본 검증 시 사이드카 LLM 호출은 문서작성 스모크에 한정(등록된 외부 서비스 사용), 지식폴더는 읽기 전용 원칙 유지.

## 5. 후속·범위 외 (기록)

- HTML 매뉴얼·스크린샷 30장·시연영상이 구 UI(100% 배율, 근거답변 명칭, 설정 단일 화면) 기준 — **매뉴얼 갱신은 별도 후속 작업**.
- T4 승인 주제(L3)는 broader=NULL이라 '기타'행(현 0건) — 후보 승인 UI에 broader 선택 추가는 후속 과제.
- validate_pack의 broader 무결성(댕글링·사이클) 경고 추가 — 후속 과제(런타임은 seen 가드로 방어).
- rhwp 종이 미리보기의 글상자 높이 렌더는 실기기 육안 1회 확인 항목(T2 risks).
