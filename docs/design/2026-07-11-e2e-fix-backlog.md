## 2. 개선계획 (일괄 수정 웨이브)

### Wave A — 지식 근거 축 복원 (치명·즉효, 최우선)
1. **F-04**: `_query_terms` 최소길이를 2자로 완화하되 2자 스톱워드(조사성 명사)만 예외 처리. 대화·ask·검색 3경로 공통화.
2. **F-01**: ask(`/api/knowledge/ask`) 검색에 대화 경로와 동일한 질의 정규화(명령어·조사 제거, 용어 분해) 적용.
3. **F-05**: `_run_document_create_skill`이 지시문 키워드로 위키 검색 → 상위 근거를 reference_texts로 주입 + 산출물에 "근거 문서" 표기. 근거 0건이면 문서에 명시.
4. **F-06**: 근거 0건 시 "자료를 확인했다"류 서술 금지 규칙(프롬프트) + 응답에 "지식폴더 근거 없음" 명시.
- 검증: askbench 8문 재실행(목표 8/8), S5 5회(목표 5/5), S4a 산출물이 실제 파일 내용 인용.

### Wave B — 문서작성 내용 규칙 (높음)
5. **F-12**: email 프롬프트에 "지시에 없는 이름·연락처·내선 생성 금지, 서명은 부서명+드림" 규칙 + few-shot 예시에서 구체 연락처 제거 + 후처리 필터(placeholder 패턴 검출 시 서명 치환).
6. **F-13/13b**: "지시에 포함된 수치·날짜는 반드시 산출물에 보존, 지시에 없는 수치 창작 금지" 규칙 + 구조 생성 후 지시 수치 보존 검사→미보존 시 1회 재생성.
7. **F-13a**: summary를 문장 단위 분리 렌더(또는 스키마에서 1문장 제약+초과분 항목화).
- 검증: email 10회(목표 placeholder 0), onePage/T2 5회(수치 보존 5/5).

### Wave C — 분류 제안 지능화 (P-2)
8. **F-07/07a**: 어휘·파일명 매칭을 전체 파일로 확장, **폴더 교차(cross-folder) 업무 후보** 생성(WORK_VOCAB+duty 토큰 → 매칭 파일의 폴더 집합을 folders로) — 폴더 승격 영역과 병렬 제시, keywords 자동 시드.

10. **F-07c**: duty 토큰을 업무 후보 시드로 사용, 폴더명 어휘 일치는 confidence medium 상한.

### Wave D — 위키 백과사전화 (P-3, 대형)
11. **F-08**: 주제 정규화 — 보강 시 기존 주제 목록을 LLM에 제공해 재사용 유도 + 유사 주제 병합(정규화 키·편집거리) + 싱글턴 주제는 상위 주제로 흡수. 목표: 주제 수 ≤ 문서수/5, 싱글턴 비율 < 30%.
12. **F-09**: **주제 페이지 백과사전 렌더(나무위키식)** — 문서 2개 이상 주제에 대해 LLM 종합 생성:
    `개요(정의) → 핵심 내용 정리(문서 내용 종합, 소제목 구조) → 경과/연표(문서 날짜 기반) → 문서별 요점 → 연관 주제`.
    보강 파이프라인에서 추출본 상위 n자를 입력으로 생성·캐시, 문서 증분 시 재생성. 링크 목록은 하단 참조로 강등.
13. **F-10**: 허브 개선 — 개요에 LLM 3~5줄 업무 요약 추가, '핵심 문서' 선정에서 양식/서식 역할 제외+보고서·계획 우선, 원본 절대경로는 툴팁으로 이동(표시는 파일명만).
14. **F-02**: 제목 규칙 — 본문유래 제목이 저품질(날짜만/조각/3자 이하)이면 파일명 사용, 주제 슬러그 생성 버그 수정.
15. **F-11**: 보강 자동화 — 색인 완료 시 잔여 전체를 배치 큐로 자동 연속 실행(이어하기), 중단 복구 시 "실패"가 아닌 "재개" 표시, 대시보드에 커버리지(요약 보유 n/전체) 노출.
- 검증: 실폴더 재보강 후 주제 페이지 표본 10개 육안 — 내용 종합 존재·가독성, 싱글턴 비율, 허브 렌더.

### Wave E — 대화중심 UX 마감 (P-1)

17. **F-15**: 일정 조회에 기간 한정어(오늘/이번 주/이번 달) 필터.

- 검증: 무세션 상태 UI 흐름 재현 2회, S7 재실행(일시 반영).

### 우선순위·규모
- **A(치명) → B → E(소형 즉효) → C → D(대형)** 순 권장. A·B·E는 국소 수정(수 시간 규모), C는 제안 엔진 확장(중형), D는 신규 생성 파이프라인(대형·LLM 비용 동반).
- 회귀 기준: 사이드카 pytest·데스크톱 vitest 그린 유지 + 본 문서의 검증 시나리오(재사용 가능한 스크립트로 보존됨) 재실행.
