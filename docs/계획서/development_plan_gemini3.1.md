# 로컬 에이전트 워크스페이스 : 공무
## 구현 가능한 개발 계획서

---

## 1. 이 프로젝트에 대한 한 줄 해석
**"단순한 챗봇을 넘어, 내 PC의 파일과 일정을 연결해 '개인 위키(지식)'를 구축하고 실제 공문서(HWPX) 산출물까지 엮어내는 오프라인 우선형(Offline-first) 지식 업무 OS"**

## 2. 전체 제품 구조 해석
이 제품은 사용자의 행위를 **수집(검색/대화) → 축적(지식폴더/파일정리) → 생산(문서작성/일정연결)** 의 사이클로 묶어내는 로컬 허브입니다.
- **라우터 역할**: '업무대화'와 '일정'이 사용자의 의도를 받아 각 기능으로 분기시키는 라우터 역할을 합니다.
- **지식 엔진**: '내 지식폴더'가 단순 디렉토리가 아닌 Graph/Wiki 형태의 백엔드 데이터베이스로 작동하며, '파일정리'는 이 DB를 최신화하는 트리거가 됩니다.
- **생산 파이프라인**: '문서작성'은 LLM이 곧바로 완성본을 찍어내는 도박이 아니라, **마크다운(MD) 추출 → 검토 → HWPX/HWTX 매핑**이라는 결정론적(Deterministic)이고 통제 가능한 파이프라인으로 작동합니다.
- **외부 연동**: 복잡한 파일 인덱싱과 검색은 검증된 외부 툴(Anything)로 위임하여 시스템의 무게를 줄이고 안정성을 확보합니다.

## 3. 전체 아키텍처 제안
로컬 Windows PC에서 구동되어야 하므로, 가벼운 프론트엔드와 AI/문서처리에 특화된 백엔드가 결합된 **데스크톱 애플리케이션 아키텍처**를 제안합니다.

*   **App Shell**: Tauri (Windows 네이티브 앱 빌드, Electron 대비 극단적으로 적은 리소스 점유)
*   **Frontend (UI/UX)**: React + TailwindCSS (복잡한 지식 그래프 시각화 및 문서 에디터 구현에 유리)
*   **Backend (Core Logic)**: Python (FastAPI) - 로컬 LLM 추론, HWPX 문서 파싱 및 조작, 지식그래프 구성에 필수적임. (Tauri의 Sidecar 형태로 Python 바이너리 패키징)
*   **Database**:
    *   메타데이터 & 일정/기록: SQLite
    *   지식 검색 (RAG): LanceDB (임베딩된 로컬 벡터 DB, 가볍고 Python 통합이 쉬움)
    *   관계형 지식 관리: NetworkX (Python 내장 그래프 라이브러리) 또는 SQLite의 브릿지 테이블 활용
*   **LLM Runtime**: Ollama (Windows 로컬 구동) 또는 llama.cpp 기반의 양자화(GGUF) 모델 직접 구동 (sLLM 7B~8B 수준)

## 4. 권장 기술스택 제안
*   **프론트엔드**: TypeScript, React, Zustand(상태관리), React Flow(지식폴더 그래프 시각화), TailwindCSS
*   **백엔드/AI**: Python 3.11+, FastAPI, LangChain/LlamaIndex(로컬 RAG 및 에이전트 라우팅), LanceDB(벡터 DB)
*   **문서 처리**: `pyhwp` (또는 HWPX의 XML 구조를 직접 파싱/조작하는 커스텀 Python 모듈), `markdown` (MD to XML 변환)
*   **패키징/배포**: Tauri, PyInstaller (Python 백엔드를 `.exe`로 독립 실행 파일화하여 배포)

## 5. 대안 기술스택 비교

| 구분 | 권장안 (Tauri + React + Python) | 대안 1 (Electron + Node.js) | 대안 2 (순수 Python - PyQt/PySide) |
| :--- | :--- | :--- | :--- |
| **장점** | Windows 리소스 점유율 최저, Python의 강력한 AI/문서 생태계 활용 가능 | 프론트/백엔드 모두 JS로 통일, 개발 인력 확보 용이 | 단일 언어로 개발, 패키징이 상대적으로 단순함 |
| **단점** | Tauri(Rust)와 Python Sidecar 간의 IPC 통신 설계 필요 | 메모리/CPU 점유율이 매우 높아 구형 공공기관 PC에서 버벅임 발생 | UI/UX 구현이 매우 제한적이고 낡아 보임. 그래프 시각화 어려움 |
| **결론** | **채택**. 공공기관 PC 성능 한계를 극복하고 AI 연산에 자원을 집중하기 위해 최적. | 기각. 로컬 LLM이 메모리를 다량 소모하므로 앱 쉘은 가벼워야 함. | 기각. 기획된 현대적 UX(지식 그래프 등) 구현 불가. |

## 6. 핵심 기능별 구현 방식

### 1) 일정
*   **구현 방식**: SQLite에 일정 데이터를 저장하되, 각 일정 레코드에 `linked_task_id`, `linked_doc_id` 메타데이터를 추가합니다.
*   **동작**: 백그라운드 스케줄러(예: Python `APScheduler`)가 일정을 감시하다가, "보고 1시간 전"에 도달하면 알림을 띄우고, 승인 시 **문서작성** 파이프라인을 자동 호출하여 초안(Markdown)을 생성해 둡니다.

### 2) 업무대화
*   **구현 방식**: LangChain 기반의 Router Chain 구조를 사용합니다.
*   **동작**: 사용자의 입력 의도를 판별하여 (1) 단순 질의응답 (2) 지식폴더 RAG 검색 (3) 특정 액션(일정 등록, 문서작성 트리거)으로 라우팅합니다. 대화의 Context는 SQLite에 세션별로 저장되며, 대화 중 발생한 유의미한 정보는 사용자 승인 후 지식폴더 DB로 Extract(추출) 됩니다.

### 3) 로컬파일/정보검색 연계
*   **구현 방식**: URI Scheme 방식(`anything://...`) 또는 OS 수준의 Process Call(`subprocess.Popen`)을 사용합니다.
*   **동작**: 앱 내 검색창에서 키워드 입력 후 엔터를 치면, Anything(애니씽) 프로그램이 해당 검색어 파라미터를 물고 외부 창으로 팝업됩니다. Anything 설치 파일은 앱 배포 시 InnoSetup 등을 통해 번들링하여 함께 설치되도록 구성합니다.

### 4) 문서작성 (핵심 파이프라인)
*   **구현 방식**: **MD → HWPX/HWTX 렌더링 엔진** 구현.
    1.  **콘텐츠 베이스**: 지식폴더 DB에서 컨텍스트를 모아 LLM이 Markdown 파일(`.md`)을 생성합니다. (재사용성을 위해 DB 저장)
    2.  **양식 파싱**: 사용자가 지정한 HWPX 파일은 압축 파일(ZIP 구조)이므로, Python의 `zipfile`과 `xml.etree.ElementTree`를 이용해 내부 `Contents/section0.xml`을 파싱합니다.
    3.  **양식 적용**: 마크다운의 구조(Heading 1, 2, List 등)를 HWPX의 특정 스타일 태그(Style ID)와 매핑하여 XML을 조작한 뒤 재압축하여 최종 `.hwpx`를 산출합니다.

### 5) 내 지식폴더 (제2의 브레인)
*   **구현 방식**: 원본 파일은 건드리지 않고, 파일의 Hash와 경로를 SQLite에 기록합니다. 해당 파일의 텍스트를 추출해 청크(Chunk)로 나누어 LanceDB에 임베딩합니다.
*   **동작 (구조화)**: 추출된 텍스트에서 LLM을 이용해 주요 엔티티(주제, 사람, 부서 등)를 뽑아내고, 이를 노드(Node)와 엣지(Edge)로 구성하여 프론트엔드의 React Flow로 그래프 형태의 위키 UI를 렌더링합니다.

### 6) 파일정리
*   **구현 방식**: 파일 시스템 이벤트 감시(Python `watchdog` 라이브러리)를 통해 타겟 폴더의 변경을 감지합니다.
*   **동작**: 새 파일이 들어오면 LLM이 내용을 빠르게 분류하여 기존 '내 지식폴더'의 어떤 노드(주제)와 연관되는지 매핑합니다. 이동 제안(UI 알림) 후, 사용자가 '승인'하면 OS 명령어(`shutil.move`)로 물리적 위치를 이동시키고 백그라운드에서 지식 DB 인덱싱을 수행합니다.

### 7) 도구
*   **구현 방식**: 플러그인 아키텍처 설계. 각 도구는 표준화된 Input/Output JSON 스키마를 가진 Python 함수(Tool)로 정의됩니다.
*   **동작**: 업무대화 에이전트가 Function Calling(Tool Calling) 기능을 이용해 필요한 도구를 자율적으로 선택하여 실행합니다.

### 8) 실행기록
*   **구현 방식**: 모든 API 호출 및 내부 서비스 레이어의 Action은 Decorator 패턴을 통해 자동으로 SQLite `audit_logs` 테이블에 적재됩니다. (timestamp, action_type, inputs, outputs, status)

### 9) 환경설정
*   **구현 방식**: 로컬의 `config.json` 또는 `settings.yaml` 파일로 관리하여, 앱이 재시작되어도 유지되도록 합니다. 지식폴더 감시 경로, LLM API 주소(내부망 서버 사용 시) 등을 관리합니다.

## 7. 기능 간 연결 구조 설계 (데이터 흐름)

```mermaid
[로컬 파일 시스템] <--감시/이동--> (6. 파일정리) --반영 후보 제안--> (5. 내 지식폴더 DB: 벡터+그래프)
                                                                    ^
[Anything (외부)] <--(키워드 전달)-- (3. 정보검색)                    | (RAG 검색 / 지식 추출)

                                       | (검색 연계)                v

                                       |                            |
                                       v                            v
                                    (7. 도구)                 [최종 HWPX 산출물]
```

## 8. MVP (Minimum Viable Product) 범위 제안
첫 번째 릴리즈(v1.0)에서는 리스크가 큰 기능은 제외하고, 핵심 가치인 **"지식화"**와 **"문서 생산"**에 집중해야 합니다.
*   **포함**: 일정(기본 CRUD 및 대화 연결), 업무대화(기본 RAG), 내 지식폴더(텍스트 인덱싱 및 마크다운 위키 보기), 문서작성(MD -> 기본 HWPX 변환), Anything 호출 연결.
*   **제외 (v1.5 이후로 연기)**: 화려한 3D 지식 그래프 시각화(리스트 형태의 위키로 대체), 복잡한 사용자 정의 HWPX 양식 완벽 호환(초기엔 2~3개의 고정 제공 템플릿만 지원), 파일정리의 '자동 이동' (초기엔 복사/반영 제안만 수행하여 원본 유실 리스크 차단).

## 9. 단계별 빌드 계획

*   **Phase 1 (Core Infra & PoC - 1달)**:
    *   Tauri + Python Sidecar 뼈대 구축
    *   Ollama 로컬 연동 및 LanceDB 텍스트 임베딩 파이프라인 구축
    *   HWPX 구조 파싱 및 MD 주입 PoC 완료
*   **Phase 2 (Knowledge & Chat - 1달)**:
    *   '내 지식폴더' 파일 감시 및 벡터 인덱싱 로직 구현
    *   '업무대화' 라우터 개발 및 지식 기반 RAG 질의응답 구현
    *   Anything 연동 버튼/기능 구현
*   **Phase 3 (Doc Pipeline & Schedule - 1달)**:
    *   '문서작성' UI 개발 (콘텐츠 베이스 에디터)
    *   일정 관리 기능 및 일정-문서작성-업무대화 연결 고리 개발
*   **Phase 4 (Polish & Security - 1달)**:
    *   파일정리(제안/승인) 로직 및 실행기록 로깅 적용
    *   Windows 실행 파일(`setup.exe`) 패키징 및 오프라인 환경 테스트 (에어갭 테스트)

## 10. 가장 먼저 검증해야 할 PoC 3~5개
1.  **MD to HWPX 변환기**: 공공기관의 핵심 산출물이므로, Markdown 기반의 콘텐츠가 HWPX의 특정 섹션에 스타일이 깨지지 않고 렌더링되는지 최우선 검증 필요.
2.  **Tauri ↔ Python 로컬 IPC 통신**: 백그라운드 Python 서버와 프론트엔드 간의 대용량 텍스트/이벤트(파일 변경 감지 등) 전송 시 성능 병목이 없는지 확인.
3.  **로컬 LLM (Ollama) 메모리 점유율 및 RAG 속도**: 실제 공공기관 평균 PC 사양(예: 8GB~16GB RAM, 내장 그래픽)에서 7B/8B 모델이 멈춤 없이 텍스트 추출과 답변을 수행할 수 있는지 검증.
4.  **Anything 외부 프로세스 호출**: 설치된 Anything을 파라미터와 함께 원활하게 팝업시킬 수 있는지 OS 레벨 검증.

## 12. 왜 이 기술 선택이 적절한지
공공기관망의 특징은 "외부 인터넷 차단(오프라인)"과 "개인 PC 리소스의 한계", 그리고 "강력한 보안(파일 유출 불가)"입니다.
*   **Tauri + Python 조합**은 Chromium 전체를 메모리에 올리는 Electron과 달리 OS 기본 웹뷰를 사용하여 **RAM을 절약**하면서도, Python의 **오프라인 NLP 및 데이터 처리 능력**을 100% 활용할 수 있게 합니다.
*   문서작성 단계를 **MD → HWPX 구조로 분리**한 것은, LLM에게 HWP 자체를 만들라고 하는 불가능한 요구를 버리고, LLM이 가장 잘하는 텍스트/구조화(MD)에만 집중시킨 뒤 렌더링은 프로그래밍 로직에 맡기는 **가장 현실적이고 성공 확률이 높은 방법**입니다.
