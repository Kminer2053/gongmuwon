# Gongmu UI/UX 개선계획 이행 결과보고

작성일: 2026-05-18

## 목표

2026-05-18 사용성 점검 결과를 기준으로 Gongmu의 실제 업무 활용성을 높이기 위한 개선계획을 이행했다. 핵심 방향은 초기 로딩 부담을 줄이고, 업무대화/지식폴더/파일찾기/실행기록의 화면 흐름을 사용자가 이해하기 쉬운 구조로 정리하는 것이다.

## 이행 결과

### 1. 초기 로딩 및 화면 전환

- 앱 초기 로딩을 shell snapshot과 deferred snapshot으로 분리했다.
- 업무대화, 지식폴더, 파일찾기, 파일정리, 실행기록 등 무거운 데이터는 해당 화면 진입 시 필요한 범위만 지연 로딩한다.
- 화면 상단에 deferred loading 상태를 작은 pill 형태로 표시해 앱이 멈춘 것처럼 보이지 않게 했다.

### 2. 업무대화

- pending assistant 메시지가 LLM 요청 payload에 섞이지 않도록 sidecar 계약을 정리했다.
- 완료된 assistant 메시지에 오래된 "잠시만 기다려 주세요"류 문구가 남지 않도록 표시 helper를 추가했다.
- 최근 응답의 GraphRAG 사용 여부, 근거 개수, 첨부 수, 연결 파일 수, 모델/provider를 업무대화 화면에 evidence 형태로 표시한다.
- 응답 실패 시 사용자에게 읽히는 한국어 실패 메시지를 보존하도록 테스트 계약을 추가했다.

### 3. 지식폴더

- 화면 구조를 `설정/상태`, `색인 처리`, `GraphRAG 검색` 탭으로 분리했다.
- 설정/상태 탭은 지식 소스 등록과 현재 상태 확인에 집중하도록 정리했다.
- 색인 처리 탭은 스캔, GraphRAG 인덱싱, 강제 재색인, 작업 진행상태, 로그 확인을 담당한다.
- GraphRAG 검색 탭은 검색, 근거 답변, 관계 보기, 출처 확인 흐름에 집중한다.
- ingestion 진행 중에는 지식 소스 재설정이나 중복 색인 액션을 막도록 UI 잠금 흐름을 검증했다.
- 구조보기, 관계 drill-down, section/table drill-down, grounded answer 생성 테스트를 보강했다.

### 4. GraphRAG 진단 및 로그 UX

- full log dump를 단순 파일 경로 안내가 아니라 `덤프 뷰어 열기` 버튼으로 확인할 수 있게 정리했다.
- 로그 뷰어는 최근 로그 개수, 이벤트, 단계, 메시지, 문서명/경로, 원본 JSON을 카드형으로 보여준다.
- ingestion 진행 상태를 진행률, 처리 수, 실패 수, 소요시간, 마지막 처리 파일 중심으로 확인할 수 있게 했다.
- 개발 중 품질 진단을 위해 상세 로그 확인 흐름을 유지했다.

### 5. 파일찾기

- 로컬 파일명 검색에서 한글 Unicode 정규화(NFC/casefold)를 적용해 파일명 검색 안정성을 높였다.
- 검색 결과의 match reason을 사용자에게 읽히는 한국어 설명으로 정리했다.
- Anything 실행 기록은 기본 화면을 덜 어지럽히도록 접힌 상세 영역으로 이동했다.

### 6. 실행기록

- `settings 쨌 failed` 같은 내부 feature/status 조합이 사용자 화면에 노출되지 않도록 실행기록 display mapper를 추가했다.
- 환경설정 저장, LLM 테스트, 업무대화 응답 실패, 파일정리 적용 등의 작업을 한국어 제목/설명으로 표시한다.
- 기존 mojibake 회귀를 막기 위한 테스트를 추가했다.

### 7. 문서작성 및 환경설정

- 업무대화 세션에서 문서작성으로 이어지는 흐름을 유지하고, 문서작성 화면에서 세션 기반 작성과 직접 작성 흐름을 구분한다.
- 환경설정의 외부 모델 provider별 저장 프로필, API key, endpoint, OpenRouter extra header, GraphRAG vector backend 설정 흐름을 검증했다.

## 남은 리스크

- 이번 목표는 사용성 개선과 회귀 방지 테스트 중심이며, NSIS/MSI 패키징 재생성은 수행하지 않았다.
- 실제 대용량 지식폴더 ingestion 품질은 문서 포맷별 parser 품질, OCR/LibreOffice fallback, HWP/HWPX parser bridge 안정화에 계속 영향을 받는다.
- 멀티모달 LLM의 이미지 이해는 현재 UI 첨부와 LLM provider API payload 계약이 완전히 맞물려야 하므로 별도 통합 검증이 필요하다.
- 현재 worktree에는 이번 목표 이전부터 누적된 변경과 생성 파일이 많으므로, 릴리스 전에는 별도 change set hygiene가 필요하다.

## 다음 수동확인 포인트

- 앱을 dev 모드로 실행해 지식폴더 탭 전환, 색인 진행 잠금, 덤프 뷰어, 구조보기 버튼이 실제 화면에서도 자연스럽게 동작하는지 확인한다.
- 업무대화에서 GraphRAG evidence 표시가 실제 모델 응답 후 적절히 갱신되는지 확인한다.
- 파일찾기에서 실제 한글 파일명을 대상으로 정확검색 체감 품질을 확인한다.
- 실행기록 패널에 내부 영문/깨진 문자열 대신 한국어 설명이 유지되는지 확인한다.

| 파일찾기 한글 검색 | API 기준 통과. `팔란티어` 검색에서 `팔란티어 파운더리 퀴즈답안.docx`가 최상위로 반환되었다. |
| 실행기록 한글 깨짐 | 통과. `쨌` 깨짐은 재현되지 않았다. 다만 `knowledge.ingest.job.run` 내부 action id가 화면에 남아 있어 추가 수정했다. |

### 자동 확인 중 발견 및 조치

- 발견: 실행기록에서 GraphRAG 인덱싱 작업이 `knowledge.ingest.job.run 작업이 완료 상태로 기록되었습니다.`처럼 내부 action id로 표시됐다.
- 조치: `knowledge.ingest.job.run`을 `GraphRAG 인덱싱 실행`으로 매핑하고 회귀 테스트를 추가했다.
- 재확인: 브라우저 새로고침 후 실행기록과 우측 작업 이력 요약 모두 `GraphRAG 인덱싱 실행 작업이 완료 상태로 기록되었습니다.`로 표시됐다.

### 자동화 한계

- 현재 Codex in-app browser의 가상 클립보드가 설치되지 않은 상태라, 브라우저 자동화에서 텍스트 입력(`fill`, `type`, DOM CUA type)이 실패했다.
- 따라서 검색어 입력이 필요한 UI 시나리오는 화면 구조 확인과 API 직접 호출로 나누어 검증했다.
