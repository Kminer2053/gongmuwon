# 전기능 통합테스트 및 사용자 경험 검증 시나리오

작성일: 2026-05-23
대상: 로컬 AI에이전트 워크플레이스 : 공무원
목적: 사용자 관점에서 제품 컨셉과 목적이 실제로 달성되는지 전 기능을 통합적으로 검증하고, 불편한 워크플로우와 개선방안을 도출한다.

---

## 1. 검증 목표

이 검증의 핵심 질문은 다음이다.

1. 사용자가 업무대화 세션을 중심으로 일정, 파일, 지식, 문서작성 작업을 자연스럽게 이어갈 수 있는가.
2. 폐쇄망/로컬 우선 제품이라는 목적에 맞게 업무엔진, 파일, 지식DB, HWPX 산출물이 로컬에서 안정적으로 동작하는가.
3. 장기 작업 중 사용자가 진행상태, 실패 원인, 취소 가능성을 이해할 수 있는가.
4. UI가 업무 흐름을 방해하지 않고, Codex 스타일의 좌/중앙/우 구조를 일관되게 유지하는가.
5. 최종적으로 외부 검수자가 “실사용 베타 수준”이라고 판단할 증거가 충분한가.

---

## 2. 검증 환경

| 항목 | 기준 |
| --- | --- |
| OS | Windows 11 x64 |
| 실행 방식 | 개발 모드: `sidecar:serve` + Vite/Tauri dev, 또는 설치패키지 |
| 업무엔진 | `http://127.0.0.1:8765` |
| UI 검증 | Codex in-app Browser 또는 Tauri 앱 직접 실행 |
| 기본 workspace | `runtime-workspace` |
| 테스트 데이터 | 작은 샘플 폴더 + 실제 업무폴더 일부 |
| LLM | Ollama 또는 OpenAI-compatible local endpoint, 외부 provider는 선택 |

주의: 현재 Codex in-app Browser의 직접 타이핑 자동화는 virtual clipboard가 없으면 제한될 수 있다. 이 경우 실제 API로 업무대화 턴을 주입하고, 화면 렌더링은 Browser로 검증한다.

---

## 3. 공통 완료 기준

각 시나리오는 아래 기준을 모두 확인한다.

- 기능 결과가 중앙 화면 또는 우측 패널에서 사용자에게 이해 가능한 형태로 보인다.
- 실패 시 `fetch fail` 같은 일반 메시지만 남지 않고 원인/단계/다음 행동이 보인다.
- 작업 진행이 필요한 기능은 우측 `작업 진행`에서 상태, 진행률, 작업 ID, 로그를 확인할 수 있다.
- 좌측 세션 rail과 상단 타이틀/업무엔진 상태가 주요 작업 중 깨지지 않는다.
- 생성/변경된 산출물은 실제 파일 경로 또는 열기 동작으로 추적 가능하다.

---

## 4. 통합 시나리오 목록

### FS-01 최초 실행과 업무엔진 상태

목적: 설치 또는 dev 실행 후 사용자가 바로 작업을 시작할 수 있는지 확인한다.

절차:

1. 앱을 실행한다.
2. 상단 제품명이 `로컬 AI에이전트 워크플레이스 : 공무원`인지 확인한다.
3. 우측 상단 업무엔진 신호등이 정상 상태인지 확인한다.
4. 신호등을 눌러 상태 팝오버가 보이는지 확인한다.
5. 새로고침을 눌러 화면이 복구되는지 확인한다.

기대 결과:

- 업무엔진 상태가 정상으로 보인다.
- 상태 팝오버가 다른 카드 밑에 가려지지 않는다.
- 새로고침 후 선택 세션/패널 상태가 비정상적으로 초기화되지 않는다.

사용성 관찰:

- “업무엔진”이라는 표현이 사용자에게 충분히 직관적인가.
- 오류 발생 시 사용자가 재시작 버튼을 찾을 수 있는가.

### FS-02 업무대화 기본 흐름

목적: 업무대화가 제품의 기본 시작점인지 확인한다.

절차:

1. 좌측 `업무대화`를 선택한다.
2. `+`로 새 세션을 만든다.
3. “오늘 해야 할 업무를 정리해줘” 같은 일반 대화를 보낸다.
4. 응답이 스트리밍 또는 pending 상태로 자연스럽게 표시되는지 확인한다.
5. 응답 완료 후 소요시간이 표시되는지 확인한다.
6. 새 메시지 후 스크롤이 마지막 대화 위치에 유지되는지 확인한다.

기대 결과:

- 사용자 입력은 우측, assistant 출력은 문서형 좌측/본문형으로 구분된다.
- 내부 reasoning이나 policy 판단이 노출되지 않는다.
- Markdown 목록, 표, 굵은 글씨가 읽기 좋게 렌더링된다.

사용성 관찰:

- 채팅 입력창 높이와 첨부 프리뷰가 대화 영역을 과도하게 가리지 않는가.
- 세부설정은 오버레이로 떠서 입력창 아래에 깔리지 않는가.

### FS-03 업무대화 도구 라우팅

목적: 자연어 요청이 일반 답변이 아니라 실제 도구 실행으로 이어지는지 확인한다.

절차:

1. 업무대화에서 “내일 오후 3시 검증 회의 일정 등록하고 지식폴더에서 프롬프트 관련 자료 찾아줘”라고 요청한다.
2. 응답에 `요청을 여러 작업으로 나누어 순서대로 처리했습니다`가 표시되는지 확인한다.
3. `일정 등록` 결과와 `GraphRAG 검색 결과`가 같은 응답 안에 보이는지 확인한다.
4. 우측 `작업 진행`에서 `work_session.turn` 작업이 완료되는지 확인한다.
5. `작업 로그 보기`를 눌러 `job.created`, `job.started`, `job.progress`, `job.succeeded` 이벤트를 확인한다.

기대 결과:

- `intent.plan -> schedule.create -> knowledge.search` 순서로 실행된다.
- 같은 세션에서 동시에 메시지를 보내도 응답 순서가 꼬이지 않는다.

사용성 관찰:

- “작업 진행” 패널이 사용자가 찾기 쉬운 위치에 열리는가.
- 도구 실행 결과와 일반 설명이 구분되는가.

### FS-04 일정 캘린더

목적: 일정 등록/수정/삭제와 세션 연결이 실제 업무 흐름에 맞는지 확인한다.

절차:

1. `일정` 화면으로 이동한다.
2. 월/주/일 보기를 전환한다.
3. 캘린더 빈 칸을 클릭해 일정을 생성한다.
4. 일정이 있는 칸 색상이 달라지는지 확인한다.
5. 긴 일정 제목이 셀 높이를 깨지 않고 축약 표시되는지 확인한다.
6. 일정 hover 또는 상세 영역에서 전체 제목을 확인한다.
7. 일정에서 연결 세션을 열거나 새 연결 세션을 만든다.
8. 일정 삭제를 수행한다.

기대 결과:

- 캘린더 셀 높이가 일정 제목 길이에 따라 들쭉날쭉하지 않는다.
- 가까운 일정은 우측 패널에서 볼 수 있다.
- 세션 연결은 중앙 업무대화와 자연스럽게 이어진다.

사용성 관찰:

- 기존 일정 클릭 후 새 일정 입력 시 이전 값이 남지 않는가.
- 월/주/일 이동 버튼 위치가 직관적인가.

### FS-05 파일찾기와 세션 파일 연결

목적: Anything 없이 자체 파일찾기로 업무대화에 파일을 연결할 수 있는지 확인한다.

절차:

1. `파일찾기` 화면으로 이동한다.
2. 검색범위에서 파일명 인덱스를 갱신한다.
3. 우측 `작업 진행`에서 `local_file_index` 작업이 표시되는지 확인한다.
4. 파일명을 검색한다.
5. 검색 결과 카드를 클릭한다.
6. 우측 미리보기에 파일명, 경로, 크기, 수정일 등이 보이는지 확인한다.
7. `경로 복사`를 눌러 토스트 피드백이 보이는지 확인한다.
8. `세션에 연결`을 눌러 현재 업무대화 세션의 연결 파일 수가 늘어나는지 확인한다.

기대 결과:

- 중앙 화면은 검색/결과에 집중하고, 상세 미리보기는 우측 패널에 표시된다.
- 경로 복사, 세션 연결 등 성공 피드백이 명확하다.

사용성 관찰:

- 검색범위/검색결과의 상하 구조가 직관적인가.
- 대량 결과에서 카드 밀도가 적절한가.

### FS-06 지식폴더 설정/스캔/GraphRAG 인덱싱

목적: 사용자가 로컬 폴더를 지식베이스로 만들 수 있는지 확인한다.

절차:

1. `내 지식폴더` 화면으로 이동한다.
2. 설정/상태 탭에서 지식 소스 폴더를 등록한다.
3. 색인처리 탭으로 이동하라는 안내가 있는지 확인한다.
4. 색인처리 탭에서 스캔 또는 GraphRAG 인덱싱을 시작한다.
5. 진행률, 현재 단계, 처리 파일명, 소요시간, 실패 수가 표시되는지 확인한다.
6. 인덱싱 중 새 지식 소스 등록/재색인이 잠기는지 확인한다.
7. 덤프 로그를 우측 덤프뷰어에서 열어본다.
8. 완료 후 색인완료 파일 수를 눌러 추출상태 화면으로 이동한다.

기대 결과:

- 설정, 색인처리, GraphRAG 검색이 명확히 분리된다.
- 작업 로그와 실패 파일 목록으로 어느 단계가 문제인지 추적 가능하다.
- 엔진이 새로고침 중 죽지 않는다.

사용성 관찰:

- 진행 애니메이션/상태바가 장시간 작업의 지루함을 줄이는가.
- 실패 메시지가 너무 길어 화면을 압도하지 않는가.

### FS-07 GraphRAG 검색과 근거 답변

목적: 지식폴더가 업무대화와 독립 화면 양쪽에서 근거 기반 답변을 제공하는지 확인한다.

절차:

1. `내 지식폴더 > GraphRAG 검색`으로 이동한다.
2. 실제 업무 질문을 입력한다.
3. 검색 결과, 근거 답변, 출처 문서, 품질 경고, 관계 보기를 확인한다.
4. 그래프 노드를 클릭해 관계 drill-down이 보이는지 확인한다.
5. 같은 질문을 업무대화에서 요청해 GraphRAG 근거가 응답에 반영되는지 확인한다.

기대 결과:

- 답변에 출처 문서가 표시된다.
- low quality/partial 문서는 경고가 보인다.
- 그래프는 드래그/스크롤/필터로 탐색 가능하다.

사용성 관찰:

- 사용자가 “왜 이 문서가 근거인지” 이해할 수 있는가.
- 그래프가 시각적으로 업무지식 지도처럼 느껴지는가, 단순 단어뭉치처럼 보이는가.

### FS-08 문서작성 HWPX 산출

목적: 세션/파일/작업설명을 기반으로 실제 HWPX 문서가 생성되는지 확인한다.

절차:

1. 업무대화 세션에 파일을 연결한다.
2. `문서작성으로 이어가기`를 누른다.
3. 세션 연결 파일 목록이 표시되는지 확인한다.
4. 파일별 활용방안을 입력한다.
5. 산출 유형을 `1페이지 보고서`로 선택한다.
6. 기본 내장 서식 또는 사용자 양식을 선택한다.
7. 작업 설명을 입력하고 `작업 시작`을 누른다.
8. HWPX 파일, 폴더, 검토용 Markdown 경로가 표시되는지 확인한다.
9. 가능하면 생성된 HWPX를 실제로 열어본다.

기대 결과:

- 더 이상 ContentBase 생성/최종 저장 요청/최종 적용 같은 복잡한 단계가 노출되지 않는다.
- HWPX 산출 파일 경로가 클릭 가능하게 보인다.
- 업무대화에서 “보고서 만들어줘” 요청도 같은 문서작성 스킬로 라우팅된다.

사용성 관찰:

- 입력 항목이 과도하게 많지 않은가.
- 보고서 양식 선택과 첨부파일 활용계획이 사용자에게 이해되는가.

### FS-09 LLM 설정과 외부/로컬 모델

목적: 모델 프로필 저장과 활성 provider가 일치하는지 확인한다.

절차:

1. `기타 환경설정`으로 이동한다.
2. 로컬, 내부 서버, 외부 모델 설정을 각각 저장한다.
3. OpenAI-compatible/OpenRouter/Ollama 등 provider별 필드가 맞게 보이는지 확인한다.
4. API key 또는 endpoint가 긴 경우 카드 영역을 넘지 않는지 확인한다.
5. 연결 테스트를 실행한다.
6. 업무대화에서 활성 provider/model이 답변에 표시되는지 확인한다.

기대 결과:

- provider별 설정이 별도로 유지된다.
- 저장 프로필 요약이 읽기 쉽다.
- 활성 provider와 저장값이 일치한다.

사용성 관찰:

- 외부 모델 설정 필드가 공식 guide 수준으로 충분한가.
- API key 보안 표시가 적절한가.

### FS-10 파일정리와 롤백

목적: 파일 변경 계열 작업이 승인/적용/롤백/작업로그로 추적되는지 확인한다.

절차:

1. `파일정리` 화면에서 테스트 폴더를 선택한다.
2. 정리 제안을 생성한다.
3. 제안 카드를 클릭해 상세를 확인한다.
4. 적용 요청을 만들고 승인한다.
5. 적용을 실행한다.
6. 우측 작업 진행에서 `fileorg.apply`가 보이는지 확인한다.
7. 되돌리기를 실행한다.
8. `fileorg.rollback` 작업과 결과 경로를 확인한다.

기대 결과:

- 파일 변경 작업은 승인과 작업로그가 남는다.
- rollback 후 같은 제안을 재요청해도 409 conflict가 나오지 않는다.

사용성 관찰:

- 사용자가 파일이 실제로 어디로 이동되는지 이해할 수 있는가.
- 제안 카드 설명이 충분한가.

### FS-11 실행기록/승인/우측 패널

목적: 우측 패널이 작업 컨텍스트를 돕는지 확인한다.

절차:

1. 승인 요청이 필요한 작업을 만든다.
2. 우측 승인 패널이 자동으로 열리는지 확인한다.
3. 작업 진행이 필요한 작업을 만든다.
4. 우측 작업 진행 패널이 자동으로 열리는지 확인한다.
5. 최근 실행 이력 카드를 눌러 상세를 확인한다.
6. 우측 패널을 접었다 펼치고 중앙 영역이 확장/복원되는지 확인한다.
7. 패널 폭 조절이 가능한지 확인한다.

기대 결과:

- 필요한 정보만 활성화되고, 불필요한 정보는 접을 수 있다.
- 작업 컨텍스트가 중앙 업무 흐름을 방해하지 않는다.

사용성 관찰:

- 아이콘만 봐도 패널 의미를 알 수 있는가.
- 우측 패널 정보가 과도하게 많지는 않은가.

### FS-12 설치패키지/폐쇄망 배포 검증

목적: 최종 배포물이 인터넷 없는 PC에서 설치/실행 가능한지 확인한다.

절차:

1. `npm.cmd run verify:all`을 통과시킨다.
2. `npm.cmd run desktop:bundle`로 NSIS 설치파일을 만든다.
3. `npm.cmd run desktop:smoke:nsis`로 자동 설치 smoke를 실행한다.
4. `npm.cmd run release:offline`으로 폐쇄망 전달용 zip을 만든다.
5. 산출물에 설치파일, sidecar bundle, release notes, 검증 문서가 포함되는지 확인한다.
6. 가능하면 별도 Windows 계정 또는 VM에서 GUI 설치/실행/삭제를 확인한다.

기대 결과:

- 설치 후 업무엔진이 자동 시작된다.
- 앱 종료/업무엔진 crash 후 복구 안내 또는 자동 재시작이 동작한다.
- NSIS uninstall 후 설치 폴더 잔여물이 명시된다.

사용성 관찰:

- 설치 경로를 모르는 사용자가 업무엔진을 별도 실행해야 하는 일이 없는가.
- 폐쇄망 PC에서 missing dependency 오류가 없는가.

## 5. 최종 판정 기준

| 판정 | 기준 |
| --- | --- |
| Pass | 기능이 목적대로 동작하고 사용자 피드백/작업로그/산출물이 명확하다 |
| Partial | 핵심 동작은 되지만 사용자가 헷갈리거나 일부 품질 한계가 있다 |
| Fail | 기능이 실패하거나, 결과가 저장되지 않거나, 원인 확인이 어렵다 |
| Blocked | 환경/모델/테스트 데이터 부족으로 확인이 불가능하다 |

릴리스 후보로 인정하려면 `FS-01`, `FS-02`, `FS-03`, `FS-05`, `FS-06`, `FS-08`, `FS-12`는 최소 Pass 또는 명확한 Partial이어야 한다.
