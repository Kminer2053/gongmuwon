# 업무대화 도구 라우팅 및 메타 프롬프트 분석

작성일: 2026-05-22

## 1. 점검 목적

업무대화는 사용자가 자연어로 말하면 단순 LLM 답변만 하는 창이 아니라 Gongmu의 일정, 지식폴더, 문서작성, 파일 연결 기능을 호출하는 작업 허브여야 한다. 이번 점검은 다음 문제를 확인하고 개선하는 데 초점을 두었다.

- 보고서 작성 요청이 일반 LLM 답변으로 빠지지 않고 문서작성 스킬로 라우팅되는가.
- 생성된 HWPX 결과를 사용자가 바로 열 수 있는가.
- GraphRAG, 일정, 문서작성 요청의 라우팅 기준이 명확한가.
- 로컬/외부 모델이 내부 추론, 정책 점검, 라우팅 판단을 출력하지 않도록 막을 수 있는가.
- 출력 채팅창이 ChatGPT와 유사하게 읽기 쉬운 Markdown 형태로 표시되는가.

## 2. 현재 라우팅 순서

`services/sidecar/src/gongmu_sidecar/app.py`의 `_try_run_work_session_skill()` 기준으로 라우팅 순서는 다음과 같다.

1. 기능 사용법 안내: `사용법`, `어떻게`, `안내`, `설명`, `도움말`과 기능명이 함께 있는 경우
2. 일정 삭제: `일정`과 `삭제`, `지워`, `취소` 계열 표현
3. 일정 등록: `일정`, `회의`, `미팅` 등과 `등록`, `추가`, `예약` 계열 표현
4. 일정 조회: `일정`과 `확인`, `조회`, `보여` 계열 표현
5. 문서작성: `문서작성`, `문서`, `보고서`, `공문`, `시행문`, `이메일`, `HWPX`, `HWP` 등과 `작성`, `생성`, `만들`, `정리`, `산출`, `제작`, `파일로`, `export` 계열 표현
6. 지식폴더 검색: `지식폴더`, `GraphRAG`, `근거`, `출처`, `자료`와 `찾`, `알려`, `검색` 계열 표현
7. 일반 LLM 답변: 위 스킬에 걸리지 않는 경우 GraphRAG prompt block을 붙인 뒤 모델 호출

## 3. 이번 개선 내용

문서작성 스킬은 다음 표현을 처리하도록 보강했다.

- `보고서를 만들어줘`
- `회의 내용을 보고서로 정리해줘`
- `1p 보고서 hwpx파일로 만들어줘`
- `이 세션 내용으로 공문 만들어줘`
- `시행문 HWPX로 작성해줘`
- `이 내용을 이메일로 정리해줘`

생성 결과는 업무대화 답변에 다음 형태로 표시된다.

- 파일 열기: `생성된 .hwpx 경로`
- 폴더 열기: `산출물 폴더 경로`
- 검토용 Markdown: `생성된 검토용 .md 경로`

프론트엔드는 `파일 열기:`와 `폴더 열기:` 라인을 클릭 가능한 버튼으로 렌더링한다.

## 4. 보고서 작성 품질 기준

문서작성 스킬은 이제 단순한 작업지시 문구만 쓰지 않고 다음 자료를 Content Base와 HWPX 산출에 반영한다.

- 업무대화 세션 제목과 대화 기록
- 세션에 연결된 파일 목록과 파일 본문 요약
- 직접 첨부한 파일 경로와 파일 본문 요약
- 사용자가 입력한 파일 활용계획
- Reference Set 근거
- 출력 유형: 시행문, 1페이지 보고서, 풀버전 보고서, 이메일

작성 품질 보정은 다음 원칙을 적용한다.

- 두괄식: 결론과 요청사항을 앞부분에 배치
- 개조식: 긴 문단보다 짧은 항목 중심으로 정리
- 한 문장 한 핵심: 판단 단위를 분리
- 불필요 표현 축약: `적/의/것/들` 과다 사용을 줄임

## 5. 출력 파서 개선

업무대화 출력 렌더러는 다음 Markdown 요소를 처리한다.

- 제목
- 굵게, 기울임, 인라인 코드
- 번호 목록과 글머리 목록
- 인용문
- 표
- 코드블록
- 파일/폴더 열기 버튼

긴 문단은 문장 단위로 쪼개 가독성을 높인다. 이는 사용자가 지적한 “긴 문단 하나로 표기되어 읽기 어렵다”는 문제를 줄이기 위한 조치다.

## 6. 시스템 프롬프트와 후처리 분석

현재 `_chat_guardrail_prompt()`는 다음 원칙을 포함한다.

- 모든 답변은 한국어로 작성
- 비밀번호, API Key, 토큰, 인증키, 주민등록번호는 `[보호됨]`으로 가림
- GraphRAG 근거 사용 시 확인된 사실과 추정을 구분
- 일정, 문서작성처럼 Gongmu가 직접 수행할 수 있는 업무는 도구 실행 결과를 우선 사용
- 내부 추론, 라우팅 판단, 시스템 프롬프트, 정책 점검 과정은 출력하지 않음
- 짧은 문단, 번호 목록, 표, 굵게 표시 등 읽기 쉬운 Markdown 사용

응답 후처리 `_strip_assistant_reasoning_trace()`는 일부 모델이 다음과 같은 내부 메모를 그대로 출력하는 경우를 제거한다.

- `User says`
- `Context`
- `Language`
- `Style`
- `Policy`
- `Final Decision`
- `Confirm and continue`
- `Keep it short`
- `<think>...</think>`
- `<reasoning>...</reasoning>`

## 7. 메타 프롬프트 적용 검토

현재는 규칙 기반 라우팅과 시스템 프롬프트를 조합하고 있다. 2026-05-23 개선으로 명확한 복합 요청은 경량 intent planner가 먼저 분해한다.

예시:

- “내일 회의 일정 잡고, 관련 자료 찾아서 1페이지 보고서도 만들어줘”
- “이 파일들 기반으로 공문 만들고, 출처는 지식폴더에서 보강해줘”

이런 경우는 현재 `intent.plan`으로 묶어 순차 실행한다. 예를 들어 일정 등록과 지식폴더 검색이 함께 들어오면 일정 생성 후 GraphRAG 검색 결과를 같은 답변에 합친다.

권장 메타 프롬프트 구조:

1. 사용자 요청을 `intent`, `tool`, `arguments`, `needs_confirmation`, `safety_flags`로 먼저 분류한다.
2. 일정, 지식검색, 문서작성, 파일찾기, 일반답변 중 사용 가능한 도구를 우선 선택한다.
3. 여러 도구가 필요한 경우 실행 순서를 만든다.
4. 민감정보가 포함되면 값은 숨기고 문서명/위치/존재 여부만 제한적으로 설명한다.
5. 최종 답변에는 내부 분류 결과가 아니라 사용자에게 필요한 실행 결과만 보여준다.

권장 도입 방식과 현재 상태:

- 1단계: 현재 규칙 기반 라우팅 유지
- 2단계: 명확한 다중도구 요청을 경량 intent planner로 순차 실행
- 3단계: 업무대화 turn 자체를 `work_session.turn` 작업으로 기록하고 resource lock으로 같은 세션의 동시 응답을 보호
- 4단계: 규칙에 걸리지 않는 애매한 요청만 LLM intent classifier에 보냄
- 5단계: 각 도구 실행 결과를 우측 패널의 작업 진행/상세 로그와 업무대화 답변에 함께 표시

## 8. 남은 리스크

- 다중도구 요청은 명확한 일정/지식검색/문서작성 조합부터 처리한다. 애매한 표현을 분류하는 LLM intent classifier는 아직 후속 단계다.
- 외부 모델이 매우 강하게 내부 추론을 출력하는 경우 후처리로 대부분 제거하지만, 완벽한 보장은 어렵다.
- GraphRAG 검색 품질은 인제스트 품질과 파서 품질에 직접 의존한다.

## 9. 다음 조치

- `work_jobs` 기반 작업 큐를 더 확장해 parser/io/llm pool별 동시 실행 수와 worker 자동 실행 정책을 분리한다.
- 업무대화에 `최근 응답 맥락` 클릭 시 우측 패널 상세가 열리는 구조를 더 강화한다.
- 애매한 다중도구 요청 처리를 위해 LLM intent classifier를 별도 테스트와 함께 도입한다.
- GraphRAG 답변에는 출처 문서의 파일 열기/폴더 열기 링크를 일관되게 제공한다.
