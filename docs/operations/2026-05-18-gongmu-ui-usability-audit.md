# Gongmu UI/UX 활용성 점검 기록

> ⚠️ **정정(2026-07-06):** ChromaDB/embedding 런타임은 이후 M-07로 제거되어 설치 패키지에 포함되지 않습니다(지식 검색은 SQLite FTS5).

작성일: 2026-05-18

## 1. 점검 목적

현재 개발된 Gongmu 기능을 실제 사용 흐름 기준으로 열어보고, 특히 사용자 UI 관점에서 미흡한 사항을 정리한다. 이번 점검은 기능 구현 완료 여부보다 "일반 업무 사용자가 앱을 켜고, 찾고, 대화하고, 지식화하고, 문서화하는 과정이 자연스러운가"에 초점을 둔다.

## 2. 점검 환경

- OS: Windows 개발 세션
- 프로젝트: `C:\Users\USER\Agent_Gongmu\Agent_Gongmu_Codex`
- UI 점검 방식: Codex in-app browser 기반 컴퓨터유즈 조작
- Frontend: `http://127.0.0.1:5173/`
- Sidecar: `http://127.0.0.1:8765/`
- Sidecar health: 정상 응답
- 실제 DB/workspace: `runtime-workspace`

## 3. 직접 확인한 화면/기능

- 업무대화: 세션 목록, 대화 이력, 일정 연결, 파일 연결, 문서작성 이동 버튼, 우측 컨텍스트 패널
- 일정: 주간 캘린더, 일정 카드 표기, 일정 입력 폼
- 로컬파일/정보검색: 내장 파일 검색, Anything 보조 검색 기록, Reference Set 생성 흐름
- 문서작성: 대화세션 기반/바로 작성, Content Base, 출력 유형, HWPX/HWTX 양식 선택
- 내 지식폴더: 지식 그래프, 설정/상태, 색인 처리, GraphRAG 검색, 로그 덤프 펼쳐보기
- 파일정리: 정리 제안, 상세보기, 적용 요청 흐름
- 실행기록: 작업 목록, 상세보기, 입력/출력 JSON
- 환경설정: 로컬/내부/외부 모델, 공급자 preset, 임베딩/Vector Store, 개인화 저장폴더
- 우측 패널: 접기/펼치기와 중앙 패널 확장 동작

## 4. 핵심 관찰 요약

앱은 주요 화면으로 이동 가능하고, 업무대화 중심의 좌/중앙/우 패널 레이아웃은 현재 방향성이 맞다. 우측 패널 접기 시 중앙 영역이 확장되는 동작도 정상이다. 파일 검색은 `pdf` 검색 기준으로 지식폴더 문서 결과를 반환했고, 지식폴더 색인 탭의 덤프 펼쳐보기 역시 실제 JSONL 로그를 표시했다.

다만 첫 진입 경험과 지식폴더/업무대화의 실사용 흐름에는 아직 품질 이슈가 남아 있다. 특히 초기 워크스페이스 로딩이 지식 문서/실행기록 API까지 한 번에 기다리면서 40초 이상 멈춘 것처럼 보였고, 업무대화 응답 카드에는 완료 후에도 `잠시만 기다려 주세요.`가 남아 사용자가 상태를 오해하기 쉽다.

## 5. UI/UX 미흡사항

### P0. 첫 화면 로딩이 전체 앱 사용을 막음

관찰:
- Sidecar `/health`는 정상인데 초기 화면은 `워크스페이스를 불러오는 중입니다...`에서 오래 머물렀다.
- `/api/knowledge/documents`는 직접 호출 시 약 42.7초 후 응답했다.
- 초기 스냅샷이 모든 목록 API를 기다리는 구조라, 일부 무거운 API 하나가 느리면 업무대화/일정 등 가벼운 화면도 열리지 않는다.

영향:
- 사용자는 사이드카가 죽은 것으로 오해한다.
- 앱 첫인상이 "멈춤"으로 느껴진다.

권장:
- 초기 로딩은 `health`, `settings`, `work-sessions`, `schedules`만 우선 로드한다.
- 지식 문서, 실행기록, 파일정리 등 무거운 목록은 화면 진입 시 lazy load한다.
- API별 timeout과 부분 성공 UI를 둔다.
- 상단 상태는 `사이드카 정상 / 지식 데이터 로딩 중 / 일부 목록 지연`처럼 분리한다.

### P1. 업무대화 완료 후 상태 문구가 정리되지 않음

관찰:
- 여러 Assistant 응답에 `잠시만 기다려 주세요.`가 최종 답변과 함께 그대로 남아 있다.
- 완료 응답에는 `응답 1.8초`처럼 시간 표기가 붙는 경우도 있지만, 과거 응답과 상태 표기가 일관되지 않다.

영향:
- 사용자는 답변이 아직 진행 중인지 완료됐는지 혼동한다.
- 장기 대화에서 "대기 중" 문구가 노이즈로 쌓인다.

권장:
- pending placeholder는 응답 완료 후 반드시 제거한다.
- 완료 상태는 `응답 1.8초`, 실패 상태는 `실패: 원인 요약`으로 단일 표기한다.
- 실행기록에는 turn 단위 이벤트를 남기되, UI 기본 화면에는 사용자에게 의미 있는 요약만 보인다.

### P1. 업무대화의 도구/RAG 연결 상태가 사용자에게 불명확함

관찰:
- 기존 대화 이력에서 모델이 "도구를 사용할 수 없다", "첨부 이미지를 볼 수 없다"는 식으로 답변한 기록이 보인다.
- 앱 관점에서는 지식폴더와 파일 연결 기능이 있지만, 모델에게 어떤 컨텍스트와 도구가 전달되는지 화면상 명확하지 않다.

영향:
- 사용자는 "GraphRAG가 연결된 업무대화"인지 "그냥 LLM 채팅"인지 판단하기 어렵다.
- 멀티모달 모델을 선택했더라도 이미지가 실제 모델 입력으로 전달되는지 확인하기 어렵다.

권장:
- 채팅 입력창 위 또는 우측 컨텍스트에 `이번 요청에 포함되는 지식: 지식폴더 N개, 연결파일 N개, 이미지 N개`를 명확히 표시한다.
- 모델 응답 하단에 `사용한 근거`, `검색 여부`, `첨부 전달 여부`, `도구 호출 여부`를 접힌 카드로 표시한다.
- 멀티모달 모델이 아니거나 이미지 전달이 비활성인 경우 전송 전 경고한다.

### P1. 지식폴더 화면 정보가 한 화면에 과밀함

관찰:
- 상단 지식 그래프, 상태 chip, 설정/상태, 색인 처리, GraphRAG 검색, 상세 데이터가 한 화면에 길게 이어진다.
- 지식 그래프 아래의 `설정/상태`, `색인 처리`, `GraphRAG 검색`은 클릭은 가능하지만 컴퓨터유즈 접근성 기준으로 버튼 role이 명확하지 않았다.

영향:
- 사용자가 "지금 무엇을 해야 하는 화면인지" 빠르게 판단하기 어렵다.
- 그래프를 보는 화면과 색인을 운영하는 화면, 검색 결과를 보는 화면의 목적이 섞인다.

권장:
- 지식폴더는 3개 작업공간으로 명확히 분리한다.
- `대시보드`: 그래프, 품질 점수, 최근 인덱싱 상태, 문제 파일 요약
- `색인/진단`: 스캔 시작, 중지, 재색인, 진행률, 실시간 로그, 덤프 뷰어
- `검색/검증`: GraphRAG 질문, 근거 문서, 관계, chunk/graph 품질 확인
- 탭은 실제 `button` 또는 `tab` role로 구현하고 선택 상태를 명확히 표시한다.

### P1. GraphRAG 진행 시간/진단 표기가 신뢰를 떨어뜨릴 수 있음

관찰:
- 17개 파일 ingestion 결과가 `소요 37599.8초 · 파일당 2211.8초`로 표기됐다.
- 실제 사용자 입장에서는 밤새 대기/중단/재시작 시간이 처리 시간에 섞인 것처럼 보인다.
- 덤프 펼쳐보기는 잘 동작하지만 긴 JSONL이 카드 안에 그대로 펼쳐져 화면을 압도한다.

영향:
- 사용자는 GraphRAG 품질/성능을 신뢰하기 어렵다.
- 어떤 단계가 느린지, 어떤 문서가 품질 문제인지 파악하기 어렵다.

권장:
- 총 경과시간과 실제 처리시간을 분리한다.
- 단계별 시간: 스캔, 파싱, 청킹, 임베딩, 그래프 저장, 검색 준비를 따로 보여준다.
- JSONL 덤프는 별도 로그 뷰어 모달로 열고, stage/filter/search/copy 기능을 제공한다.
- `품질 낮음`, `본문 부족`, `metadata fallback` 같은 문제 파일을 별도 큐로 모아준다.

### P1. 파일검색은 동작하지만 결과보다 Anything 기록이 먼저 과하게 보임

관찰:
- `pdf` 검색 시 지식폴더 문서 결과가 정상적으로 반환됐다.
- 다만 검색 전 화면에서는 오래된 Anything 실행 이력이 길게 노출되어, 기본 기능인 내장 파일찾기보다 보조 기능이 더 크게 보인다.
- 일부 파일명은 자모 분리 형태로 보였다.

영향:
- 사용자는 "이 화면이 파일 검색인지 Anything 히스토리인지" 혼동할 수 있다.
- 한국어 파일명 표시 품질이 낮아 보일 수 있다.

권장:
- 기본 화면은 검색 입력, 인덱스 상태, 최근/추천 결과 중심으로 단순화한다.
- Anything 기록은 `외부 고급검색 기록` 접힌 섹션으로 내린다.
- 파일명/경로는 Unicode NFC 정규화 후 표시한다.
- 검색 결과에서 `세션에 연결`을 누르면 연결 수가 업무대화 버튼 옆에 즉시 반영되게 한다.

### P2. 문서작성 화면에 과거 템플릿 개념과 신규 출력 유형이 함께 보여 혼동됨

관찰:
- 화면에는 `보고서형/회의자료형/검토메모형`과 `시행문/1페이지 보고서/풀버전 보고서/이메일`이 함께 보인다.
- 사용자가 기대한 최종 출력 유형은 후자에 가깝다.

영향:
- 사용자는 어떤 선택이 실제 HWPX 출력 구조를 바꾸는지 알기 어렵다.

권장:
- 문서작성의 핵심 선택은 `출력 유형`으로 통합한다.
- 기존 `보고서형/회의자료형/검토메모형`은 내부 skill/prompt preset으로 숨기거나 고급설정으로 이동한다.
- 대화세션에서 넘어온 경우 `이 세션에서 문서작성` 흐름을 최상단에 고정한다.

### P2. 실행기록은 상세하지만 아직 업무자가 읽기 어려움

관찰:
- 목록에는 `knowledge.ingest.job.run`, `work_session.turn.completed` 같은 raw 이벤트명이 섞여 있다.
- 상세보기는 입력/출력 JSON을 그대로 보여준다.
- `상세보기` 버튼이 매우 많이 반복된다.

영향:
- 개발자에게는 유용하지만 일반 업무자에게는 의미를 해석하기 어렵다.

권장:
- 기본 목록은 `지식폴더 색인 완료`, `업무대화 응답 실패`, `문서 저장 적용`처럼 완전한 한국어 문장으로 통일한다.
- raw JSON은 `개발자 정보` 접힌 영역으로 이동한다.
- 기능/상태/기간 필터를 상단에 제공한다.
- 같은 turn의 message created/completed 이벤트는 하나의 카드로 묶는다.

### P2. 환경설정은 기능은 많지만 정보 밀도가 높음

관찰:
- 외부 모델 공급자, API Key, Base URL, OpenRouter 옵션, 임베딩, Vector Store, 개인화 저장폴더가 한 화면에 모여 있다.

영향:
- 사용자가 무엇부터 설정해야 하는지 알기 어렵다.
- API Key 입력과 연결 테스트 결과가 공급자별 프로필 저장 상태와 연결되어 보이는지 추가 확인이 필요하다.

권장:
- `모델 연결`, `지식/임베딩`, `개인화`, `고급`으로 섹션을 나눈다.
- 현재 활성 프로필과 저장된 공급자별 프로필을 카드로 요약한다.
- 연결 테스트 결과에는 요청 provider/model/base_url을 함께 표시한다.

## 6. 기능별 현재 활용성 판단

| 기능 | 현재 활용성 | 판단 |
| --- | --- | --- |
| 업무대화 | 보통 | 세션 중심 구조는 좋지만 RAG/도구/첨부 전달 상태가 불명확하다. |
| 일정 | 양호 | 주간 일정 표기와 등록 흐름은 이해 가능하다. 월/일 뷰와 편집 안정성은 추가 수동 점검 필요. |
| 파일찾기 | 양호 | 내장 검색은 동작한다. Anything 기록 과노출과 한글 파일명 정규화가 필요하다. |
| 문서작성 | 보통 | Content Base/HWPX 흐름은 보이나 출력 유형과 템플릿 개념이 혼재한다. |
| 지식폴더 | 부분 양호 | 핵심 기능은 보이나 운영 화면, 진단 화면, 검색 화면의 정보 설계가 더 필요하다. |
| 파일정리 | 부분 양호 | 승인형 흐름은 보이나 실제 사용자 목적과 정리 결과 설명이 더 필요하다. |
| 실행기록 | 부분 양호 | 상세 추적은 강하지만 사용자용 요약과 개발자용 raw 로그를 분리해야 한다. |
| 환경설정 | 부분 양호 | 외부 모델 설정 항목은 갖췄지만 공급자별 프로필 상태와 입력 가이드가 더 선명해야 한다. |

## 7. 우선 개선 순서

1. 초기 로딩 구조 개선: 핵심 화면 먼저 열고 무거운 목록은 lazy load한다.
2. 업무대화 응답 상태 정리: pending 문구 제거, 응답시간/실패 원인/근거 사용 여부를 일관 표기한다.
3. 지식폴더 IA 재정리: 대시보드, 색인/진단, 검색/검증을 명확히 분리한다.
4. GraphRAG 로그 뷰어 개선: stage별 진행률, 실시간 로그, 문제 파일 큐, 덤프 모달을 제공한다.
5. 파일찾기 기본 흐름 강화: 내장 검색을 최상위로 두고 Anything은 접힌 보조 섹션으로 이동한다.
6. 실행기록 사용자화: raw event를 한국어 요약으로 매핑하고 JSON은 개발자 정보로 숨긴다.
7. 문서작성 출력 유형 정리: 시행문, 1페이지 보고서, 풀버전 보고서, 이메일 중심으로 재구성한다.

## 8. 추가 수동 확인이 필요한 항목

- Tauri 네이티브 창에서 메뉴바 확대/축소, 폴더 선택, 파일 열기, Anything 실행 등 OS 연동 기능
- 실제 로컬/외부 멀티모달 모델에 이미지가 base64 또는 파일 참조로 전달되는지 여부
- GraphRAG 검색 결과가 업무대화 요청에 자동으로 포함되는지 여부
- 폐쇄망 설치본에서 sidecar 자동 시작/재시작, ChromaDB/embedding runtime 포함 여부
- 장시간 ingestion 중 스캔/재색인 버튼 비활성화와 중지/재개 흐름

## 9. 결론

현재 Gongmu는 "업무대화 중심의 로컬 업무 워크스페이스"라는 방향은 분명해졌다. 좌측 세션, 중앙 작업, 우측 컨텍스트라는 기본 레이아웃도 맞는 방향이다. 다음 품질 단계의 핵심은 기능 추가보다 `느린 데이터 로딩을 UI 전체와 분리`, `업무대화에 실제 포함되는 지식/파일/첨부를 명확히 표시`, `지식폴더를 운영자용 진단 도구처럼 정리`하는 것이다.

특히 첫 로딩과 GraphRAG 색인/진단 화면은 제품 신뢰도에 직접 영향을 준다. 이 두 영역을 먼저 정리하면 이후 문서작성, 파일찾기, 개인화 학습 기능의 체감 품질도 함께 좋아질 가능성이 높다.
