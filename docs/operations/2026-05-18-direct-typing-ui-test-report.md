# 2026-05-18 직접 타이핑 기반 UI 기능 테스트 결과

## 테스트 목적

in-app browser에서 virtual clipboard가 없어 `fill/type` 자동화가 실패하는 상황을 확인했다. 대신 실제 키보드 입력에 가까운 `click + keypress` 방식으로 주요 입력 UI를 점검했다.

- 앱 URL: `http://127.0.0.1:5173`
- 업무 엔진: `http://127.0.0.1:8765/health`, `status=ok`
- 입력 방식: in-app browser computer-use 기반 `click`, `keypress`
- 제약: 한글 IME 조합 입력은 자동화하지 못해 ASCII 입력으로 기능 흐름을 검증했다.

## 직접 타이핑 테스트 대상 선정

직접 타이핑이 기능 흐름을 좌우하는 입력 UI를 우선순위로 선정했다.

- 세션 검색: 좌측 세션 목록 필터링
- 업무대화 입력창: 메시지 작성 및 Enter 제출
- 내장 파일찾기: 파일 검색어 입력 및 검색 실행
- GraphRAG 검색: 지식 검색어 입력 및 검색 실행
- 문서작성: 문서 제목/목적 입력

환경설정 API 키/endpoint 입력은 실제 사용자 설정 오염 위험이 있어 저장 테스트에서 제외했다.

## 발견사항

### 1. 문서작성 `문서 목적` 라벨 연결 오류

`getByLabel("문서 목적")`로 접근하면 실제 목적 textarea가 아니라 `보고서형` 입력칸으로 포커스가 이동했다.

확인 증거:

- 목적 textarea placeholder: `대화세션에서 이어받은 작업 목적, 또는 바로 작성할 문서 개요를 적습니다.`
- `문서 목적` label 접근 시 확인된 placeholder: `보고서형`

영향:

- 마우스로 클릭하는 사용자는 큰 문제를 못 느낄 수 있다.
- 키보드 접근성, 스크린리더, 자동화 테스트, 라벨 기반 focus에서는 잘못된 필드에 입력될 수 있다.

권장 수정:

- `문서 목적` label의 `htmlFor`와 textarea `id`를 1:1로 다시 연결한다.
- 문서작성 폼의 모든 label/input id 중복 여부를 점검한다.

### 2. 파일검색은 Enter 제출이 더 안정적으로 확인됨

검색어 입력 후 버튼 클릭 시 한 차례 결과 갱신이 즉시 확인되지 않았고, 입력칸에서 Enter 제출 시 결과가 정상 표시됐다.

영향:

- 실제 사용자는 검색 버튼을 누를 가능성이 높으므로 버튼 submit 흐름도 재점검이 필요하다.

권장 수정:

- 검색 버튼 클릭과 Enter 제출이 동일 handler를 타는지 확인한다.
- 검색 중 상태와 결과 갱신 상태를 명확히 표시한다.

### 3. GraphRAG 검색 버튼 활성화 타이밍

첫 시도에서는 검색 버튼이 비활성으로 판정됐으나, 화면 전환 후 데이터 로딩이 끝난 뒤 재시도하면 정상 동작했다.

영향:

- 사용자가 탭을 열자마자 입력/검색할 때 버튼이 잠깐 비활성일 수 있다.

권장 수정:

- 로딩 중이면 `지식 데이터 로딩 중`과 버튼 비활성 사유를 검색 영역 안에 직접 표시한다.
- 로딩 완료 후 버튼 활성 상태가 즉시 반영되는지 확인한다.

## 결론

핵심 직접 타이핑 흐름은 대부분 정상 동작한다. 다만 문서작성의 label 연결 오류는 접근성/자동화/키보드 사용자 관점에서 실제 수정 대상이다. 파일검색 버튼 클릭과 GraphRAG 로딩 타이밍은 치명적 블로커는 아니지만 사용자가 “눌렀는데 반응이 애매하다”고 느낄 수 있어 다음 UI 안정화 작업에 포함하는 것이 좋다.
