# 로컬 에이전트 워크스페이스 "공무" — 개발 계획서

> 공공기관 사무업무자를 위한 로컬 업무 에이전트 시스템
> 작성일: 2026-04-19

---

## 1. 프로젝트 한 줄 해석

> **"공공기관 내부망 Windows PC에서, 파일 찾기·지식 축적·문서 생산·업무 대화가 하나의 흐름으로 이어지는 로컬 우선 개인 지식업무 허브"**

핵심은 챗봇이 아니라 **"일정이 트리거, 지식폴더가 중심, 문서가 산출물"인 업무 파이프라인**이다.

---

## 2. 전체 제품 구조 해석

### 2.1 본질 해석

이 프로젝트는 세 가지 축으로 구성된 하이브리드 제품이다.

```
[업무 실행 축]       일정 → 업무대화 → 문서작성 → 실행기록
      ↕                       ↕                       ↕
[지식 축적 축]       로컬검색(애니씽) → 내 지식폴더 → 파일정리
      ↕                       ↕                       ↕
[플랫폼 축]          도구 ↔ 환경설정 ↔ 승인 흐름
```

- **업무 실행 축**: 휘발성 작업. 오늘·이번주의 task 중심
- **지식 축적 축**: 영속성 자산. 장기 축적되는 제2의 브레인
- **플랫폼 축**: 위 두 축을 엮는 접착제

### 2.2 기능별 비중 해석


| 기능     | 비중    | 해석                              |
| ------ | ----- | ------------------------------- |
| 내 지식폴더 | ★★★★★ | 제품 차별화 핵심. Obsidian + RAG + 그래프 |
| 문서작성   | ★★★★★ | 실제 업무 산출물. 2단 구조(콘텐츠→양식)        |
| 파일정리   | ★★★★☆ | 지식폴더 피더. 단독 기능 아님               |
| 업무대화   | ★★★★☆ | 라우터. 중심이 아닌 허브                  |
| 일정     | ★★★☆☆ | 진입점. 단순 캘린더 아님                  |
| 로컬검색   | ★★★☆☆ | 외부 연계(애니씽). 직접 구현 X             |
| 도구     | ★★☆☆☆ | 플러그인 확장 포인트                     |
| 실행기록   | ★★☆☆☆ | 투명성 보장 (감사 요건)                  |
| 환경설정   | ★☆☆☆☆ | 인프라                             |

## 3. 전체 아키텍처 제안

### 3.1 레이어 구조

```
┌─────────────────────────────────────────────────────────────┐
│ [L1] UI Layer — Desktop App (Windows 네이티브 느낌)         │
│   React/Svelte + Tauri Shell                                │
│   좌측 메뉴 · 중앙 작업공간 · 우측 참고/승인 패널              │
├─────────────────────────────────────────────────────────────┤
│ [L2] Orchestration Layer — 업무 흐름 라우터                  │
│   Task Router · Session Manager · Approval Gateway          │
│   Intent Classifier (대화 → 기능 매핑)                       │
├─────────────────────────────────────────────────────────────┤
│ [L3] Feature Services Layer — 기능별 도메인 서비스            │
│   Schedule · Chat · Doc · Knowledge · FileOrg · Tools       │
├─────────────────────────────────────────────────────────────┤
│ [L4] Intelligence Layer — AI 추론 엔진                       │
│   Local LLM (Ollama) · Embedding · RAG · Reranker           │
│   Prompt Templates · Tool-use Framework                     │
├─────────────────────────────────────────────────────────────┤
│ [L5] Storage Layer — 데이터 저장소                           │
│   SQLite (메타) · LanceDB/Chroma (벡터) · 파일시스템 (원본)   │
│   Markdown Vault (지식폴더) · Audit Log (실행기록)           │
├─────────────────────────────────────────────────────────────┤
│ [L6] Integration Layer — 외부 연계                           │
│   애니씽 런처 · HWPX 변환기 · Outlook(선택) · MCP 도구        │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 데이터 흐름 (핵심 유스케이스)

```
[문서작성 흐름]
사용자 요청
  → 업무대화 라우터 (Intent: "문서작성")
  → 참고자료 수집 (지식폴더 RAG + 로컬검색 결과 + 대화 맥락)
  → 콘텐츠 베이스 생성 (Markdown 중간 산출물)
  → 양식 선택 (내장 템플릿 or 사용자 HWPX)
  → 양식 매핑 엔진 (콘텐츠 → 필드)
  → 사용자 검토/편집
  → 승인 게이트
  → 최종 문서 산출 + 콘텐츠 베이스는 지식폴더 환류
  → 실행기록
```

```
[지식 축적 흐름]
파일정리 스캔
  → 변경 감지 (modified since last scan)
  → 분류 제안 (지식폴더 구조 기반)
  → 지식 반영 후보 생성 (요약·엔티티·주제)
  → 승인 게이트
  → 지식 문서 업데이트 + 그래프 갱신
  → 실행기록
```

### 3.3 프로세스 구성 (Windows 로컬)

```
[Tauri 앱 프로세스]
  ├─ UI 렌더러 (WebView2)
  └─ Rust Core (파일 I/O, 권한, OS 연동)

[Python 백엔드 사이드카]  ← stdio/localhost로 Tauri와 통신
  ├─ FastAPI (내부 localhost:8765)
  ├─ LlamaIndex/LangChain (RAG 파이프라인)
  └─ HWPX 변환 워커

[Ollama 서비스]  ← 별도 Windows 서비스
  └─ 로컬 LLM 추론 (Llama 3.1 8B / Qwen 2.5 7B)

[외부 프로세스]
  └─ 애니씽 (Anything) — 링크 방식 기동
```

## 6. 핵심 기능별 구현 방식

### 6.1 일정 (Schedule)

**컨셉**: 업무 연결 캘린더 (Calendar as Workflow Trigger)

**데이터 모델**

```yaml
ScheduleItem:
  id: uuid
  title: str
  datetime: range
  type: "회의" | "보고" | "행사" | "기한" | "일반"
  linked_tasks: [task_id]       # 연결된 작업
  linked_docs: [doc_id]          # 연결된 문서
  linked_sessions: [session_id]  # 연결된 대화
  pre_actions: [action]          # 일정 전 자동 준비
  post_actions: [action]         # 일정 후 자동 처리
```

**구현 방식**

- FullCalendar.js로 월/주/목록 뷰
- SQLite `schedules` 테이블 + `schedule_links` 관계 테이블
- APScheduler로 `pre_actions` 트리거 (예: 회의 30분 전 자료 취합)
- ICS 파일 가져오기/내보내기로 Outlook 연동 우회

**핵심 연결 흐름**

```
[회의 일정 등록]
  → "회의자료 초안 준비" 액션 선택
  → 일정 시간 30분 전 자동 실행:
     1. 지식폴더에서 "회의 주제" 관련 문서 RAG 조회
     2. 콘텐츠 베이스 초안 자동 생성
     3. 사용자에게 알림 "초안 준비 완료, 검토하시겠어요?"
```

### 6.2 업무대화 (Chat Router)

**컨셉**: 업무 요청 라우터 (일반 챗봇 X)

**핵심 구조: Intent Classification + Action Suggestion**

```
[사용자 입력]
  → Intent 분류기 (Local LLM 판단)
     ├─ "문서 작성 요청" → 문서작성 기능으로 이동 제안
     ├─ "파일 찾기" → 로컬검색(애니씽) 제안
     ├─ "지식 질문" → 지식폴더 RAG 답변
     ├─ "일정 등록" → 일정 기능으로 이동 제안
     └─ "일반 질의" → 직접 답변

[답변 생성 시]
  → RAG 파이프라인 (지식폴더 + 참조 파일)
  → 답변 + 출처 메타데이터
  → 답변 하단에 액션 버튼:
     [문서 초안으로 보내기] [지식폴더에 반영] [일정에 연결]
```

**세션 관리**

- 세션별 참조 컨텍스트 유지 (파일, 폴더, 지식주제)
- 세션 자체를 "작업 단위"로 저장 → 재개 가능
- SQLite `sessions`, `messages`, `session_refs` 테이블

**구현 포인트**

- 스트리밍 응답 (Ollama `/api/chat` stream)
- 출처 표시: 답변 문장 단위로 청크 ID 매핑
- 컨텍스트 윈도우 관리: 8K 넘으면 이전 메시지 요약 압축

### 6.3 로컬파일/정보검색 (애니씽 연계)

**구현 범위 엄수**: 검색 엔진 직접 구현 안 함. 진입점/활용점만 구현.

**구현 방식**

```
1. 원클릭 설치 시 애니씽 번들 설치 (NSIS 인스톨러)
   - Everything.exe + ETP 서버 모드 활성화

2. "로컬파일/정보검색" 메뉴
   ├─ [애니씽 열기] 버튼 → shell:execute("Everything.exe")
   ├─ 간이 검색 입력창 (ETP 프로토콜로 결과 미리보기만)
   └─ 활용 연결 패널

3. 검색 결과 활용 연결
   - 결과 파일 우클릭 → 컨텍스트 메뉴
     ├─ "업무대화로 보내기" (파일 경로 컨텍스트 주입)
     ├─ "문서작성 참고자료로 추가"
     ├─ "지식폴더에 반영 후보로 등록"
     └─ "지식폴더에 편입"
```

**ETP(Everything Protocol) 활용**

- 애니씽의 TCP 기반 검색 프로토콜 사용
- 본 시스템은 클라이언트로만 동작 (검색 엔진 X)
- 결과 받아서 "활용 패널"만 렌더링

### 6.4 문서작성 (2단 구조)

**절대 원칙**: 콘텐츠 베이스 → 양식 적용의 **2단 분리**

**파이프라인**

```
[1단계: 콘텐츠 베이스 생성]
  사용자가 문서 목적 선택 (보고서/기획서/공문/검토서/회의록 등)
  → 참고자료 수집기
     ├─ 지식폴더 RAG (주제 관련 지식 문서)
     ├─ 로컬파일 참고 (사용자 첨부)
     ├─ 대화 세션 맥락
     └─ 일정 정보 (연결된 경우)
  → LLM 콘텐츠 베이스 생성기
     ├─ 문서 유형별 프롬프트 템플릿 적용
     ├─ 섹션 구조 (YAML frontmatter + Markdown 본문)
     └─ 출처 각주 자동 삽입
  → 결과: content_base.md (재사용 가능 자산)

[2단계: 양식 선택 및 적용]
  양식 선택 UI
     ├─ 내장 템플릿 (보고서/공문/검토서 각 3~5종)
     ├─ 사용자 업로드 HWTX/HWPX
     └─ 과거 사용 양식 재사용
  → 콘텐츠 → 양식 매핑 엔진
     ├─ 필수 항목 자동 인식 (수신처/제목/본문/담당자 등)
     ├─ 섹션 ↔ 양식 필드 LLM 매칭
     └─ 누락 항목 사용자 확인
  → 편집 화면 (양식 적용 후 미리보기)
  → 승인 게이트
  → 최종 출력 (HWPX/DOCX/PDF)

[환류]
  content_base.md → 지식폴더에 "문서 자산"으로 저장
  → 다음 유사 문서 작성 시 참고 가능
```

**양식 매핑 엔진 구현**

- 내장 `anthropic-skills:public-doc-to-hwpx` 스킬 활용
- HWPX 구조 분석 → 필드 슬롯 탐지 → 콘텐츠 섹션과 의미적 매핑
- LLM 프롬프트:
  ```

  [콘텐츠 섹션]: {content_sections}

저장: NetworkX → SQLite (nodes, edges 테이블)

```

**(3) 지식 검색 (RAG)**

```
질의 → 임베딩 → LanceDB top-20
     → 리랭커(bge-reranker) top-5
     → 그래프 확장 (관련 노드 +2 hop)
     → 컨텍스트 구성
     → LLM 답변 + 출처 표시
```

**(4) 살아있는 지식**

- 원본 변경 감지 → "반영 후보" 큐에 등록

## 8. MVP 범위 제안

### 8.1 MVP 원칙

**"지식폴더 중심 업무 한 사이클이 닫히는 최소 제품"**

한 사이클 = `대화로 물어봄 → 지식폴더에서 답변 → 콘텐츠 베이스 생성 → HWPX 출력`

### 8.2 MVP 포함 (Must)


| 기능         | MVP 수준                                       |
| ---------- | -------------------------------------------- |
| **일정**     | 기본 CRUD + 월/주 뷰. 자동실행은 제외                    |
| **업무대화**   | 세션, RAG 답변, 출처 표시. 액션 제안 1종(문서작성 연결)         |
| **로컬검색**   | 애니씽 실행 + 결과 파일 "업무대화로 보내기"만                  |
| **문서작성**   | 콘텐츠 베이스 생성 + 내장 템플릿 3종(보고서/공문/검토서) + HWPX 출력 |
| **내 지식폴더** | 폴더 등록 + 원본 인덱싱 + 주제 페이지 수동 생성 + RAG          |
| **파일정리**   | 변경 감지 + 지식 반영 후보 알림 (자동 정리 X)                |
| **실행기록**   | 기본 타임라인 + 승인 이력                              |
| **환경설정**   | 폴더/모델/애니씽 경로                                 |
| **도구**     | OCR, 문서요약, HWPX 변환만                          |


### 8.3 MVP 제외 (Next)

- 지식 그래프 시각화 (목록+링크만)
- 일정 자동실행 파이프라인
- 플러그인 SDK
- Outlook 연동
- 내부망 서버 모드
- 사용자 HWPX 양식 업로드 (내장 템플릿만)

### 8.4 MVP 성공 지표

1. 공공기관 사용자가 **10분 내** 지식폴더 1개 등록 + 파일 10개 인덱싱 가능

3. 콘텐츠 베이스 → HWPX 변환 **1분 내** 완료
4. 외부 네트워크 호출 **0회** 검증 (감사)

## 12. 왜 이 기술 선택이 적절한가

### 12.1 Tauri + Python Sidecar

**공공기관 제약과의 정렬도**

- Electron(100MB+)은 내부망 배포에 부담. Tauri(15MB)는 이메일 전달도 가능
- Rust 코어는 메모리 안전성 보장 → 감사 대응 유리
- Python은 LLM/RAG 생태계 최강. 분리하면 UI와 독립 업데이트 가능

### 12.2 Ollama + Qwen 2.5 7B

**대체 불가능한 세 가지 이유**

1. **한국어 성능**: 공개 벤치(KoBest, KLUE) 기준 동급 최강
2. **8GB RAM 동작**: 공공기관 표준 PC에서 돌아감
3. **Windows 서비스 통합**: 원클릭 설치 후 백그라운드 상주

### 12.3 LanceDB

- 서버 불필요 → 내부망 정책 회피
- 파일 기반 → 백업/이동 SQLite 수준으로 간단
- 컬럼 기반 → 메타데이터 필터링 성능 탁월

### 12.4 Markdown + YAML 지식 포맷

- **이식성 최강**: Obsidian/Logseq/Notion 모두 호환
- **감사 용이**: 사람이 직접 읽음 → 블랙박스 아님
- **미래 대비**: 제품 폐기되어도 사용자 자산 보존

### 12.5 콘텐츠 베이스 2단 구조

- 양식이 바뀌어도 콘텐츠 재사용
- 양식 적합화 공수 LLM이 담당 → 사용자는 콘텐츠에 집중
- 콘텐츠 베이스 자체가 **지식 자산으로 환류** → 누적 효과

## 13. 최종 추천안

### 13.1 원샷 요약

> **"Tauri 데스크톱 셸 + Python RAG 사이드카 + Ollama 로컬 LLM + 마크다운 기반 지식볼트"** 조합으로, **19주 7 Phase**로 구축한다. **MVP는 지식폴더 한 사이클 완결**에 집중하고, 그래프 시각화·일정 자동실행·플러그인 SDK는 v2로 미룬다. **PoC 5종은 본개발 착수 전 2주 이내에 끝낸다.**

### 13.2 즉시 실행 Top 5

1. **PoC 1 (LLM 한국어 품질) 즉시 착수** — 이게 성공해야 나머지 설계가 유효
2. **PoC 2 (HWPX 매핑) 병렬 진행** — 실패 시 제품 정체성 재정의 필요
3. **Phase 0 스캐폴딩** — Tauri+Python+Ollama 번들이 원클릭으로 설치되는 가장 작은 앱
4. **샘플 지식폴더 구조 완성** — 공공기관 실제 업무 1종을 기준으로 `00_원본/10_주제/20_프로젝트` 구조 검증
5. **파일럿 사용자 2명 확보** — 스펙 쓴 분과 별개로, 실제 사용할 공무원 조기 참여

### 13.3 핵심 설계 원칙 5 (개발 중 흔들리면 복귀할 북극성)

1. **지식폴더가 주인공**이다. 대화·문서·파일정리는 지식폴더를 먹이거나 꺼내 쓴다.
2. **콘텐츠 베이스와 양식은 반드시 분리**한다. 절대 바로 HWPX 생성 X.
3. **외부 호출 0회가 기본값**이다. 위반하려면 사용자 동의 + 기록 필수.
4. **승인 없는 쓰기 작업 없음**. 자동화는 제안까지, 적용은 사용자 결정.
5. **실행기록은 기능이 아니라 인프라**다. 모든 작업이 자동으로 남는다.

### 13.4 예상 비용·공수

- **개발자 1명 기준 4.5개월**
- 하드웨어: 개발 테스트용 Windows PC (RAM 16GB 권장) 1대
- 외부 비용: 없음 (오픈소스 스택만 사용)
- 초기 파일럿: 2~3명 공무원 협업 필요
